# Тестер USR-Modbus — дизайн (PZP-02, Подзадача 2)

> Дата: 2026-07-16
> Клиент: ПЗП. Статус: дизайн утверждён, готов к плану реализации.
> Предшественник: [Дизайн коннектора USR-Modbus (Подзадача 1)](2026-07-03-usr-modbus-connector-design.md), раздел 10 «Проверки на месте (вход для Тестера)».

## 1. Назначение

Тестер — полевой инструмент инженера-наладчика для ввода шлюза **USR-IO424T-EWR** в
эксплуатацию на конкретном ТПА. Инженер запускает его на ноутбуке, физически подключённом к
шлюзу по Modbus TCP, дёргает контакты и двигает узлы машины и по реакции шлюза:

1. проверяет связь (IP / порт `0x1076` / Unit ID);
2. сопоставляет физические дискретные входы (DI) семантическим ролям сигналов и определяет
   полярность контактов;
3. калибрует аналоговые входы (положение шнека/плиты, мВ → мм);
4. получает готовый, валидный `connector.json`;
5. проверяет собранную конфигурацию на реальном цикле литья через тот же автомат состояния,
   что и боевой коннектор.

Тестер закрывает «вход для Тестера» из дизайна Подзадачи 1: реальный TCP-порт и Unit ID,
соответствие DI↔роль и полярность, диапазоны аналогов и калибровка `scale`/`offset`.

## 2. Инварианты

- **Read-only.** Тестер использует только читающие функциональные коды Modbus (0x01–0x04)
  через `IModbusReader` из Core. Никакой записи в шлюз ни в одном режиме — включая режим
  верификации (MQTT не публикуется, только отображение).
- **Переиспользование Core, без дублирования.** Тестер не содержит собственного Modbus-слоя,
  типов конфига, декодеров и автомата состояния — всё берётся из
  `Wintime.Connector.UsrModbus.Core`. Тестер добавляет только UI, логику калибровки и writer
  конфигурации.
- **Совместимость выхода.** `connector.json`, который пишет Тестер, обязан читаться боевым
  коннектором без изменений. Гарантируется круг-трипом через `Core.ConfigLoader` перед
  записью.

## 4. Главное меню (режимы)

Приложение — меню-ориентированный TUI. Пункты:

1. **Подключение.** Ввод IP, порта, Unit ID (значения инженер берёт из web-конфигуратора
   шлюза). Автопроверка: устанавливает TCP-соединение, делает пробное чтение DI и читает
   версию MCU (`0x00B4` software, `0x00B5` hardware) — это подтверждает, что на адресе
   действительно USR, а не посторонний Modbus-узел. Показывает измеренный round-trip.
   Параметры связи хранятся в сессии и используются остальными режимами.

2. **Live-монитор сырых сигналов.** Диагностический инспектор, не требующий конфигурации.
   Живая таблица, обновление ~2–4 Гц:
   - DI1–DI4 — `0x0020..0x0023`, FC02 (уровень входов);
   - реле (обратное чтение) — `0x0000..0x0003`, FC01;
   - AI1/AI2 — `0x0058`/`0x0059`, FC04, значение в мВ;
   - температура PT100 — `0x0050`, FC04, `(raw − 10000) / 100`;
   - счётчики импульсов — `0x0040..0x0043`, FC04, `uint16` с обработкой wrap.

   **Кнопочные регистры `0x0030..0x0033` не читаются** — они самоочищаются при чтении
   (см. Modbus Read Spec §3.2).

3. **Мастер: собрать `connector.json`.** См. раздел 5.

4. **Проверить `connector.json` (live-верификация).** См. раздел 6.

5. **Выход.**

## 5. Мастер сборки `connector.json`

Пошаговый визард. Итог — валидный `connector.json`.

- **Шаг 0 — профиль.** Выбор `SingleNode` / `TwoNode`. Профиль задаёт набор обязательных
  ролей (`ConnectorProfileValidator.RequiredRoles`).

- **Шаг 1 — маппинг дискретных входов.** Для каждой дискретной роли:
  1. Тестер снимает базовый уровень всех DI;
  2. просит инженера активировать соответствующий сигнал («сомкни форму», «дай впрыск» и
     т.п.);
  3. по изменению уровня определяет, какой DI сменил состояние, и **направление** изменения;
  4. определяет полярность: если активация даёт переход 1→0, контакт нормально-замкнутый
     (NC) → `Invert = true`; иначе `Invert = false`;
  5. предлагает привязку DI→роль, инженер подтверждает или переназначает.

  Обязательные роли берутся по профилю (SingleNode: `Injection`, `EjectorFwdReached`;
  TwoNode: дополнительно узел 2). Опциональные роли (`MouldClosed`, `MouldOpened`,
  `EjectorBwdReached`, `Reject`) инженер маппит по желанию; непривязанные DI можно оставить
  как публикуемые поля без роли (`SignalRole.None`) или не включать вовсе.

- **Шаг 2 — калибровка аналоговых входов.** Для каждой аналоговой роли
  (`InjectionPosition`, `MoldPosition`; для TwoNode — `InjectionPosition2`):
  1. выбор физического регистра (`0x0058` / `0x0059`);
  2. выбор режима калибровки:
     - **Две точки (Scale + Offset):** инженер ставит узел в положение A, вводит значение в
       мм (с HMI ТПА), Тестер усредняет мВ по серии опросов; затем положение B → мм; линейная
       аппроксимация даёт `Scale = (mmB − mmA) / (mvB − mvA)` и
       `Offset = mmA − Scale × mvA`.
     - **Одна точка (только Scale, ратиометрия):** `Offset = 0`, `Scale = mm / мВ` в одном
       известном положении (или прямой ввод коэффициента, напр. 0.0225 / 0.045 из
       Modbus-спеки).
  3. показ результата и остаточной ошибки, возможность пересемплировать;
  4. `Unit = "mm"`.

- **Шаг 3 — параметры опроса.** Дефолты `pollIntervalMs = 500`, `timeoutMs = 1000`,
  при желании правятся. Секция `stateMachine` — дефолты Core (в JSON можно не писать).

- **Шаг 4 — запись.** Сборка `ConnectorConfig` → `ConnectorProfileValidator.Validate` →
  сериализация в JSON схемы Core → **круг-трип: `ConfigLoader.LoadFromJson` прочитывает
  собранный JSON обратно** (если не читается — ошибка сборки, не пишем файл) → предпросмотр →
  запись в `config/<alias>.json` (путь/alias вводит инженер, дефолт — этот каталог). После
  записи — предложение перейти в режим верификации по только что записанному файлу.

**Схема JSON.** Пишется ровно то, что читает `Core.ConfigLoader`: секции `device`, `profile`,
`registers[]` (поля `name`, `address` в hex-строке, `access`, `role`, `scale`, `offset`,
`unit`, `invert`), опционально `stateMachine`. Enum-значения — через `JsonStringEnumConverter`
в том же регистре, что принимает `ConfigLoader` (camelCase-совместимо, без учёта регистра).

## 6. Режим live-верификации

Отдельный пункт меню (не хвост мастера) — принимает путь к **любому** `connector.json`: по
умолчанию только что собранному, но допускается правленный руками или ранее развёрнутый файл.
Развязка верификации и генерации — намеренное решение: проверять можно конфиг из любого
источника.

Поток:
1. загрузка через `Core.ConfigLoader.LoadFromFile` — тот же путь, что использует боевой
   коннектор;
2. конструирование `ConnectorEngine(config, () => new NModbusReader(host, port, timeoutMs))`;
3. подписка на `ConnectorEngine.StateUpdated`, запуск `RunAsync` в фоне;
4. живой рендер `MachineState`: связь, `MachineMode` (Auto / Idle / Alarm / Offline),
   `CycleCounter`, подушка (`Fields["cushion"]`), текущие позиции шнека/плиты, время с
   последнего цикла, `LastCycleCompletion`.

Инженер гоняет реальный цикл литья и наблюдает, что коннектор распознаёт впрыск, инкрементирует
счётчик и корректно считает подушку. **MQTT не публикуется** — Тестер только отображает
состояние, поэтому режим read-only и без побочных эффектов.

## 7. Тестируемость (xUnit)

Вся расчётная логика выносится в чистые функции/сервисы, тестируемые без железа и консоли
(Modbus — за фейковым `IModbusReader`):

- линейная аппроксимация по двум точкам и расчёт по одной точке;
- определение полярности/`Invert` из снимков DI «до/после» активации;
- определение, какой DI изменил состояние (и обработка неоднозначности — изменилось 0 или >1
  входов);
- writer `connector.json` и его круг-трип через `Core.ConfigLoader.LoadFromJson`.

Modbus-I/O (`NModbusReader`) и Spectre-рендер — тонкие адаптеры, юнит-тестами не покрываются.
Расчётные сервисы получают уже снятые значения (мВ, снимки DI) и не зависят от транспорта.

## 8. Границы (не входит)

- Запись в шлюз (настройка порта/Unit ID/режимов) — делается через штатный web-конфигуратор
  USR, не Тестером (инвариант read-only).
- Изменение семантики автомата состояния или схемы конфига — это зона Core (Подзадача 1).
- MQTT-публикация, интеграция с Control API — зона Host боевого коннектора.
- Автоскан портов/Unit ID — не делаем: параметры инженер берёт из web-конфигуратора, связь
  подтверждается чтением версии MCU.

## 9. Открытые технические детали (уточняются при реализации на железе)

- Точный TCP-порт (`0x1076`) и Unit ID конкретного шлюза — вводятся инженером.
- Порядок байт при чтении `ushort` (свап) — сверить с примером температуры `0x0692` = 1682
  (Modbus Read Spec §7); реализация чтения уже в `Core.NModbusReader`/декодерах.
- Реальные диапазоны аналогов и число точек усреднения при калибровке — подобрать на месте.
