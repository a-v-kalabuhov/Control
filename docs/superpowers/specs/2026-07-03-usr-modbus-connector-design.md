# Дизайн: коннектор USR-Modbus (PZP-02)

**Дата:** 2026-07-03, ред. 2026-07-12
**Статус:** принят к реализации
**Задача:** PZP-02 (для ПЗП). Съём сигналов с ТПА без программного интерфейса — через
IoT-шлюз USR-IO424T-EWR (сухие контакты + аналог → Modbus TCP) и публикация телеметрии
в Wintime Control.

> **Изменения относительно первой редакции.** Первая редакция описывала флаговый
> `ModeResolver`, режим `manual`, цикл как полную 4-фазную последовательность и мгновенный
> замер подушки. По итогам обследования ТПА (циклограмма BM180-MT) и уже разработанного ядра
> `UsrConnector` эти решения **заменены** (см. §2). Ядро принято как фундамент; данная спека
> покрывает **верхний слой — интеграцию с Control** и промоушен кода в структуру решения.

---

## 1. Контекст

Часть ТПА (BORCHE BM180-MT с контроллером KEBA 2980 у ПЗП) не имеют OPC/REST. Контроллер
релейными выходами и роботным интерфейсом отдаёт дискретные сигналы фаз и аналог положения;
они заводятся на шлюз **USR-IO424T-EWR** (4 DI сухой контакт + 2 AI 0–10 В), который отдаёт их
по **Modbus TCP**. Коннектор опрашивает шлюз (read-only, 500 мс), вычисляет состояние машины и
публикует телеметрию в MQTT в контракте Control.

Обследование, карта регистров и предметная область зафиксированы в
`docs/tasks/usr424-modbus-connector/`: `CYCLOGRAM_SINGLE_NODE.md`, `PROJECT_SUMMARY.md`,
`USR-IO424T-EWR_Modbus_Read_Spec.*`. Ядро коннектора — `.../code_new/UsrConnector`.

## 2. Готовое ядро (фундамент, семантику не трогаем)

`UsrConnector.Core` (+ ~25 тестов) — зрелый нижний слой, **read-only**, без знания о транспорте.
Ключевые решения (полностью — в его `STATE_MACHINE.md`; здесь фиксируем как принятые):

- **Якорь цикла — фронт `Injection ↑`** (не `MouldClosed`: форма смыкается и вручную). На фронте:
  `CycleCounter++` (счётчик износа формы), `Mode=Auto`, открыть окно цикла.
- **Штатное завершение — фронт `EjectorFwdReached ↑` (E1)**; защёлкиваются подушка/длительность/reject.
- **Подушка = минимум `InjectionPosition` за окно цикла** (мгновенный замер неверен из-за
  дискретизации опроса). Публикуется при любом исходе.
- **Режимы `MachineMode` = `Idle/Auto/Alarm/Offline`** (без `manual`). Idle и Alarm
  взаимоисключающие: был впрыск → возможен только Alarm (E1 не пришёл за таймаут); не было →
  только Idle.
- **Адаптивные таймауты**: среднее по последним N успешным auto-циклам × коэффициент
  (`alarmCoef`/`idleCoef`), seed 300 с, сброс статистики при простое > 15 мин (переналадка).
- **Offline** — N подряд неудачных опросов; активный цикл → `Interrupted`.
- **Семантические роли** (`SignalRole`): логика видит сигналы только через роли, не адреса.
  Профили `singleNode` / `twoNode`; обязательны роли `Injection` + `EjectorFwdReached`.
- **Публичный контракт `MachineState`**: `{ Mode, CycleCounter(long), LastCycleCompletion,
  TimestampUtc, Fields }`, где `Fields` — словарь (`cushion`, `cushion2`, `lastCycleDurationMs`,
  `reject`, `moldPosition`, `injectionPosition`, доп. поля шаблона).
- `ConnectorEngine` — оркестрация **одного** устройства (`PeriodicTimer`, событие
  `StateUpdated`, переподключение к Modbus по ошибке).

**Инвариант:** семантика автомата/ролей/контракта меняется только синхронно (код + тесты +
`STATE_MACHINE.md`). Наша работа — слой **над** контрактом, ядро по существу не правим.

## 3. Область работ (Подзадача 1 — интеграция с Control)

Верхний слой, который ядро осознанно оставило снаружи. **Это и есть остаток задачи.**

**Входит:**

1. Промоушен и переименование ядра в структуру решения (`Wintime.Connector.UsrModbus`).
2. Мост `MachineState` → телеметрия Control (маппинг режима/счётчика/полей).
3. Мультиустройство: по одному `ConnectorEngine` на ТПА, одновременный опрос (`Task.WhenAll`).
4. `IMachineSource`: список ТПА из Control (API) или из файла; связка `ConnectorAlias` →
   локальная конфигурация устройства.
5. MQTT-паблишер в контракт Control; подавление публикации в `Offline`.
6. Кроссплатформенность (Linux-контейнер) + Docker.
7. Тесты (xUnit) на слой маппинга и источник машин.

**Не входит:** изменение семантики ядра; **Тестер (Подзадача 2)** — отдельной спекой (калибровка
порогов/инверсии/полярности, реальный порт/Unit ID, соответствие DI↔роль); изменения схемы
Control; двухузловой профиль на реальном железе (роль `twoNode` в ядре есть, обкатка отложена);
глубокий анализ аварий по событию KEBA (нужна документация оператора — отложено).

## 4. Контракт Control (мишень маппинга)

Проверено по коду (`Infrastructure/Handlers/*`, `ConnectorsController`).

- **Топик:** `control/imm/{immId}/telemetry`.
- **Payload:** `{ "timestamp": ISO|unix, "mode": string, "sensors": { имя→значение } }` —
  все три поля обязательны, `sensors` непуст.
- **`mode`** ∈ `auto/manual/idle/alarm` (нормализуется trim+lower); `offline` не шлём.
- **Счётчик циклов** Control считает сам: сенсор шаблона с `ParameterType=="cycleCounter"`,
  парсит `int`, фиксирует цикл при **изменении** значения в режиме `auto`.
- **Валидация сенсоров** по `ParameterName`/`ParameterType` шаблона; неописанные — игнорируются;
  невалидный `Required` — отклоняет сообщение. **Имена выходных полей обязаны совпадать с
  `ParameterName` шаблона ТПА в Control.**
- **Offline** Control выводит по `Template.DeviceTimeoutSeconds` — достаточно перестать публиковать.
- **Список ТПА:** `GET /api/connectors/{connectorType}/machines`, заголовок `X-Api-Key`
  (сверяется с `ConnectorApiKey`). Отдаёт активные ТПА с `Template.ConnectorType==connectorType`:
  `{ ImmId, ImmName, ConnectorAlias, TemplateConfig=JsonConfig }`.
- **Slug:** `usr-modbus` (значение `Template.ConnectorType`).

## 5. Мост MachineState → телеметрия

Слой в Host, чистая функция + тонкий паблишер.

**Режим:** `Idle→"idle"`, `Auto→"auto"`, `Alarm→"alarm"`, `Offline→` **не публиковать**
(Control сам выведет offline по таймауту). `manual` ядром не производится.

**Сенсоры** (ключи = `ParameterName` шаблона; соответствие — на операторе, как в Keba):

- `cycleCounter` ← `MachineState.CycleCounter` (тип `cycleCounter`/`int`). Счётчик монотонный
  от 0; в диапазоне `int` при 500 мс — десятки лет, переполнение непрактично. Публикуем целым.
- `cushion` (`float`) ← `Fields["cushion"]`; аналогично `cushion2`, `lastCycleDurationMs`,
  `moldPosition`, `injectionPosition`.
- `reject` (`boolean`) ← `Fields["reject"]`.
- доп. поля шаблона (`ExtraFields`) — прозрачно.

Значения сериализуются в строку в инвариантной культуре (double с точкой), как ждёт валидатор.

**Публикация произвольных сырых сигналов — чистой конфигурацией (без правок ядра).** Регистр с
`role: none` (дефолт) `RoleMapper` кладёт в `ExtraFields[name]`, ядро пробрасывает в
`MachineState.Fields`, Host публикует как `sensors[name]`. Оператор добавляет в шаблон ТПА сенсор
с `ParameterName = name` — сигнал сохраняется. Так публикуется любой дискрет/аналог, не
участвующий в автомате (`MouldClosed`, `MouldOpened`, `EjectorBwdReached`, PT100-температура и
т.п.). **Сигналы-якори** (`Injection`, `EjectorFwdReached`) несут семантическую роль и в `Fields`
не попадают; если нужны сырыми — завести второй регистр на тот же адрес с `role: none` и
отдельным именем (оба читаются одним батч-запросом), не трогая ядро. По умолчанию v1 сырые
дискреты не публикует (режим/счётчик/подушка/положения информативнее) — включается по мере
надобности через конфиг + шаблон.

## 6. Мультиустройство и источник ТПА

- Один шлюз USR (4 DI + 2 AI) = **ровно один ТПА**. Host ведёт много ТПА, **по одному
  `ConnectorEngine` на устройство**, все опрашиваются одновременно (`Task.WhenAll`, как
  `KebaPollingWorker`). `ConnectorEngine.StateUpdated` → маппинг → MQTT publish.
- `IMachineSource` (api|file), по эталону Keba: `ControlApiClient` дергает
  `/api/connectors/usr-modbus/machines`; `FileMachineSource` читает локальный список.
- **Связка `ConnectorAlias` → устройство: каталог файлов по alias.** Один файл на ТПА —
  `config/<alias>.json` в формате `samples/connector.json` **без изменений** (`device`
  host/port/unitId/poll/timeout, `profile`, `stateMachine` коэффициенты, `registers` — адреса,
  роли, `scale/offset`, инверсия, калибровка). Каталог монтируется в контейнер как volume; Тестер
  (Подзадача 2) пишет калибровку пофайлово. Host для каждого ТПА из Control грузит
  `config/<alias>.json`; ТПА без файла — предупреждение в лог и пропуск. Формат ядра не меняется.
- Один MQTT-клиент на процесс (publish потокобезопасен).

## 7. Структура решения (переименование)

По эталону Keba/OpcUa, в `Sources/Connectors/`:

- `Wintime.Connector.UsrModbus.Core` — из `UsrConnector.Core` (переименование неймспейсов),
  generic-ядро, семантика заморожена.
- `Wintime.Connector.UsrModbus.Core.Tests` — из `UsrConnector.Core.Tests`.
- `Wintime.Connector.UsrModbus` — Host (`BackgroundService` + Generic Host): `IMachineSource`
  (`ControlApiClient`/`FileMachineSource`), `MqttPublisher`, `StateTelemetryMapper`,
  оркестратор мультиустройства, загрузка локальной конфигурации по alias.

Порядок: сначала перенести/переименовать ядро и добиться **build + test зелёными** (в среде
генерации не компилировалось — без SDK), затем добавлять Host-слой инкрементально.

## 8. Кроссплатформенность и Docker

- Таргет `net9.0` (без `-windows`; в отличие от Keba, привязки к Windows нет). NModbus `3.0.*`
  и MQTTnet — кроссплатформенные.
- `Dockerfile` (multi-stage sdk→runtime); локальная конфигурация и appsettings — volume;
  MQTT-брокер, URL Control и `X-Api-Key` — через env. Работает в локальной сети предприятия.

## 9. Тестирование (xUnit, TDD)

- **Ядро** — существующие ~25 тестов (исполняемая спецификация автомата): перенести, держать
  зелёными; при переименовании не менять семантику.
- **Слой маппинга** (`StateTelemetryMapper`) — новые тесты: режим→строка, `Offline`→подавление,
  `CycleCounter`→строка, `Fields`→`sensors` (инвариантная культура, bool/float), пустой набор
  полей, соответствие имён.
- **`IMachineSource`** — парсинг ответа API/файла, связка alias→устройство, пропуск ТПА без
  локальной секции.
- Калибровка и поведение на железе — вне юнитов (зона Тестера, Подзадача 2).

## 10. Проверки на месте (вход для Тестера)

Реальный TCP-порт (`0x1076`) и Unit ID (`0x00B1`, обычно 17); соответствие DI↔роль и полярность
контактов (порядок в `connector.json` под фактический монтаж); диапазоны аналогов и калибровка
`scale` (ratiometric, ≈22.5 мм/В впрыск, ≈45 мм/В смыкание); свободные контактные группы реле
KA23/KA27/KA34/KA24; удержание усилия смыкания (замер DO3 — справочно). Всё это формирует и
проверяет Тестер (Подзадача 2).
