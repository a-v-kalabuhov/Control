# Дизайн: коннектор USR-Modbus (PZP-02)

**Дата:** 2026-07-03
**Статус:** принят к реализации
**Задача:** PZP-02 (для ПЗП). Съём сигналов с ТПА без программного интерфейса —
через IoT-шлюз USR-IO424T-EWR (сухие контакты → Modbus TCP) с публикацией
телеметрии в Wintime Control.

---

## 1. Контекст и цель

Часть ТПА (напр. BORCHE с контроллером KEBA у ПЗП) не имеют OPC/REST-интерфейса, но
контроллер выводит релейными (беспотенциальными) выходами дискретные сигналы фаз цикла и
имеет аналоговые выходы. Эти контакты заводятся на IoT-шлюз **USR-IO424T-EWR**, который
отдаёт их по **Modbus TCP** (RTU-семантика внутри). Коннектор опрашивает шлюз, преобразует
сырые регистры в понятные Control параметры (**режим работы** и **счётчик циклов**), калибрует
аналоги и публикует телеметрию в MQTT в стандартном формате Control.

«Прямое подключение к сухим контактам» из бэклога — обманчивое название: физически коннектор
работает не с контактами напрямую, а с Modbus TCP шлюзом (USR-IO424 или совместимым).

Эталон реализации — существующие коннекторы `Wintime.Connector.Keba` (OPC DA) и
`Wintime.Connector.OpcUa` в `e:\Projects\Control\Sources\Connectors\`.

## 2. Область работ

**Входит (Подзадача 1 — коннектор):**
- Кроссплатформенный сервис-коннектор, опрашивающий один и более шлюзов USR по Modbus TCP.
- Двухэтапная обработка: stateless-декод регистров → stateful-синтез `mode` + `cycleCounter`.
- Публикация телеметрии в MQTT в контракте Control.
- Получение списка ТПА из Control (API или файл), локальная конфигурация декодирования.
- Юнит-тесты (xUnit) на чистую логику.
- Docker-образ и пример конфигурации.

**Не входит (отдельные спеки):**
- **Подзадача 2 — Тестер** шлюза: утилита для калибровки (снятие серии показаний,
  сопоставление с наблюдаемыми данными, подбор порогов/инверсии DI, определение реального
  TCP-порта и Unit ID). Здесь только закладываем, что калибровочные параметры живут в
  локальном конфиге, чтобы тестер мог их формировать.
- Изменения схемы/кода приложения Control. Контракт Control используется как есть.
- Глубокий анализ аварий, построение кривой цикла, SPC по подушке — как точки расширения.

## 3. Контракт Control (мишень для выхода коннектора)

Проверено по коду `Wintime.Control.Infrastructure/Handlers/*` и `ConnectorsController`.

**Топик:** `control/imm/{immId}/telemetry` (ровно 4 сегмента, `immId` — Guid).

**Payload (обязательные поля):**
```json
{ "timestamp": "2026-07-03T10:00:00Z", "mode": "auto", "sensors": { "...": "..." } }
```
- `timestamp` — ISO-строка или Unix-секунды.
- `mode` — строка; канонические значения `auto` / `manual` / `idle` / `alarm`
  (`ImmMode`, нормализуется trim + lower). `offline` не шлём.
- `sensors` — непустой объект `имя→значение` (значения сериализуются в строку).

**Счётчик циклов.** Control (`CycleProcessingHandler`) считает циклы **сам**: находит сенсор
шаблона с `ParameterType == "cycleCounter"`, парсит его как `int` и фиксирует завершённый цикл
при **изменении** значения счётчика в режиме `auto`. Значит коннектор обязан публиковать
**монотонно возрастающее целое** — по одному инкременту на завершённый цикл.

**Валидация сенсоров.** `ValidateTelemetryDataHandler` сверяет каждый сенсор с шаблоном по
`ParameterName` и `ParameterType` (`string`/`float`/`int`/`boolean`/`cycleCounter`); сенсоры,
которых нет в шаблоне, молча игнорируются; невалидный обязательный (`Required`) сенсор
отклоняет всё сообщение. **Имена выходных полей коннектора обязаны совпадать с
`ParameterName` шаблона ТПА в Control.**

**Offline.** Выводится Control по таймауту `Template.DeviceTimeoutSeconds` — коннектору
достаточно перестать публиковать при недоступности шлюза.

**Список ТПА.** `GET /api/connectors/{connectorType}/machines`, заголовок `X-Api-Key`
(сверяется с `ConnectorApiKey` в конфиге Control). Возвращает активные ТПА, у которых
`Template.ConnectorType == connectorType`: `{ ImmId, ImmName, ConnectorAlias, TemplateConfig }`
(`TemplateConfig` = распарсенный `Template.JsonConfig`).

**Slug коннектора:** `usr-modbus` (значение `Template.ConnectorType`).

## 4. Архитектура

Новый проект `Wintime.Connector.UsrModbus` в `Sources/Connectors/`, по эталону Keba, но
**кроссплатформенный** (таргет `net9.0`, **без** `AddWindowsService` — OPC DA привязка к
Windows у Keba здесь отсутствует). Основной способ запуска — Docker-контейнер.

```
appsettings/env ─► Host (Generic Host, BackgroundService)
                     │
   IMachineSource ───┤  список ТПА: ControlApiClient (/api/connectors/usr-modbus/machines)
   (api | file)      │                либо FileMachineSource
                     ▼
   локальный конфиг ─► по ConnectorAlias: параметры устройства, карта регистров,
   (registers/rules)   калибровка, пороги, правила
                     │
                     ▼  на каждый ТПА — свой Poller (Task.WhenAll, все опрашиваются одновременно)
   ┌─────────────────────────────────────────────────────────────────┐
   │ Poller (PeriodicTimer pollIntervalMs)                            │
   │  1. IModbusReader читает ReadBlock[]        ← NModbus TCP        │
   │  2. TagDecoder: regs → TagSnapshot          ← ЭТАП 1 (stateless) │
   │  3. StateEngine: snapshot → mode+counter    ← ЭТАП 2 (stateful)  │
   │  4. MqttPublisher.PublishTelemetry(immId, mode, sensors)         │
   └─────────────────────────────────────────────────────────────────┘
```

**Переиспользуется из прототипа `docs/tasks/usr424-modbus-connector/code`** (перенести в проект,
доработав неймспейсы/DI): `Config`/`ConfigLoader` (модель и загрузка), `BatchPlanner`
(группировка регистров в минимум запросов), `TagDecoder` (stateless-декод), `IModbusReader` +
`NModbusReader` (Modbus TCP, абстракция транспорта), каркас `Poller`. Прототип уже держит
несущую границу stateless/stateful и батчинг — переносим её как инвариант.

**Инварианты прототипа сохраняются:** read-only (никаких FC записи); граница stateless/stateful
(память между опросами — только в stateful-слое); батчинг смежных регистров; метка времени —
после ответа устройства; прикладной код зависит от `IModbusReader`, не от NModbus.

## 5. Этап 1 — stateless-декод

Из карты регистров (локальный конфиг, по мотивам `registers.json` прототипа) на каждый опрос
строится `TagSnapshot`:
- **4×DI** (FC 0x02 `0x0020…0x0023`) → `bool`, с опциональной инверсией на тег.
- **AI1 PlatenDistance** (FC 0x04 `0x0058`, мВ) → `float`, `scale/offset` (калибровка).
- **AI2 Cushion** (FC 0x04 `0x0059`, мВ) → `float`, `scale/offset` (калибровка).

Специфика USR (из `USR-IO424T-EWR_Modbus_Read_Spec.md`): разные FC по типу данных;
big-endian снимается NModbus; аналог напрямую в мВ (`scale=0.001` → В); Unit ID (обычно 17) и
TCP-порт (регистр `0x1076`, нестандартный) уточняются на железе; кнопочные регистры
`0x0030…0x0033` не читать (самоочистка). Значения аналогов «дрожат» — при необходимости
сглаживание добавляется в stateful-слое, не в декодере.

## 6. Этап 2 — флаги, детекция цикла, режим

Stateful-движок на каждый ТПА. Работает над потоком `TagSnapshot`, хранит состояние.

### 6.1. Флаги
- **Фазовые (из DI):** `moldClose`, `injection`, `ejection`, `moldOpen`.
- **Пороговые (из AI):** конфигурируемый список сравнений аналога с порогом → `bool`
  (напр. `moldFullyClosed = platenDistance < X`). Пороги — в конфиге (калибруются тестером).
- **Цикловые (внутри текущего цикла):** `cycleStart`, `cycleEnd`, `injectionSeen`,
  `ejectionSeen`.

### 6.2. Детекция цикла
**Цикл = полная последовательность фаз** `закрытие → впрыск → выталкивание → открытие`.
Конечный автомат отслеживает прохождение фаз по фронтам DI (с мёртвым временем против
дребезга). Завершение полной последовательности:
- `cycleCounter++` (монотонный, с обработкой — счётчик коннектора, не регистр устройства);
- длительность цикла = время между `cycleStart` и `cycleEnd`;
- если за цикл наблюдались **и** `injection` **и** `ejection` — цикл **полный**, его
  длительность добавляется в **скользящее окно N** (дефолт 20, конфиг). Иначе цикл
  **холостой** (в среднее **не** входит).

### 6.3. ModeResolver (чистая функция)
По духу `ImmEffectiveStatus.Resolve` в Control — чистая, полностью юнит-тестируемая функция от
{флаги, тайминги, статистика, параметры конфига}. `avg` — среднее по окну полных циклов.
Приоритет сверху вниз:

1. **alarm** — если `alarmMultiplier > 0` **и** длительность текущего (незавершённого) цикла
   `> alarmMultiplier × avg`. (`alarmMultiplier` = 0 «выкл» либо вещественное ≥ 2.)
2. **manual** — последний завершённый цикл холостой (не было `injectionSeen && ejectionSeen`).
3. **idle** — если `idleMultiplier > 0` **и** время от `cycleEnd`
   `> idleMultiplier × avg`. (`idleMultiplier` = 0 «выкл» либо вещественное ≥ 1.)
4. **auto** — иначе (циклы идут нормально).

**Cold start.** `avg` считается **только по полным 4-фазным циклам**; холостые исключены. Пока
нет ни одного полного цикла, `avg` не определён → правила idle/alarm по длительности
**неактивны** (движок не выдаёт idle/alarm из тайминга до появления базовой статистики).

### 6.4. Формирование payload
`sensors` собирается из выходных полей (имена — из локального конфига, совпадают с
`ParameterName` шаблона Control):
- `cycleCounter` (int, монотонный) — обязателен;
- `cushion` (float, калиброванный);
- `platenDistance` (float, калиброванный);
- сырые состояния фаз DI как отдельные `boolean`-сенсоры (`moldClose`/`injection`/`ejection`/
  `moldOpen`) — публикуются; лишние пользователь убирает из шаблона Control.

`mode` — результат ModeResolver. `timestamp` — момент после ответа устройства.

## 7. Конфигурация

**Локальный файл** (правится руками, монтируется в контейнер как volume). Секция на каждый
ТПА, **ключ = `ConnectorAlias`** (значение приходит из Control вместе со списком ТПА):
- подключение: `host`, `port`, `unitId`, `pollIntervalMs` (дефолт 500), `timeoutMs`,
  `reconnectDelayMs`, `maxBatchGap`;
- карта регистров: адрес, `access` (FC), `rawType`, `count`, `scale`/`offset`, инверсия, `unit`;
- пороговые флаги по аналогам (имя флага → тег + оператор + порог);
- параметры правил: `window` (N), `idleMultiplier`, `alarmMultiplier`, имена фазовых сигналов,
  мёртвое время дребезга;
- **имена выходных полей** сенсоров (обязаны совпадать с `ParameterName` шаблона в Control).

**appsettings/env** (не секретится в образ): MQTT (`BrokerHost`/`Port`/`TopicTemplate`),
источник ТПА (`Source` = `api|file`, `SourcePath`, `ApiKey`), интервалы переподключения.

Схема Control не меняется. Согласованность «имя выходного поля ↔ `ParameterName` шаблона» —
на операторе (та же связка, что в Keba/OPC-UA).

## 8. Кардинальность и конкурентность

- Один шлюз USR-IO424 (4 DI + 2 AI) = **ровно один ТПА**. Коннектор ведёт много шлюзов.
- **Все ТПА опрашиваются одновременно** — по одному `Poller` на устройство, запуск через
  `Task.WhenAll` (как `KebaPollingWorker.PollMachineAsync`). Полностью async, без блокировок.
- Один MQTT-клиент на процесс, publish потокобезопасен (MQTTnet).
- Каждый `Poller` самостоятельно переподключается к Modbus при ошибке (единичное TCP-соединение,
  переоткрытие только по сбою; последовательный опрос внутри устройства).

## 9. Кроссплатформенность и Docker

- Таргет `net9.0`, C# latest, nullable enabled. Без Windows-специфики (в отличие от Keba).
- NModbus `3.0.*` (кроссплатформенный, Modbus TCP), MQTTnet — как в существующих коннекторах.
- `Dockerfile` (multi-stage build на `mcr.microsoft.com/dotnet/sdk` → `runtime`), конфиг и
  `appsettings` монтируются volume'ом; MQTT/URL Control — через env. Работает в локальной сети
  предприятия рядом с Control и брокером.

## 10. Тестирование (xUnit, TDD)

Чистая логика покрывается юнит-тестами до реализации:
- `ModeResolver` — таблица истинности флагов + тайминги + множители (0/выкл, граничные ≥1/≥2),
  cold start (нет полных циклов), холостой цикл → manual, застревание → alarm.
- `CycleStateMachine` — полная/неполная последовательность фаз, дребезг (мёртвое время),
  учёт полных циклов в окне и исключение холостых, монотонность `cycleCounter`.
- `BatchPlanner`, `TagDecoder`, парсер конфига — переносятся из прототипа с тестами.
Калибровка и поведение на реальном железе — вне юнит-тестов (зона тестера, Подзадача 2).

## 11. Точки расширения (вне v1)

Глубокий анализ аварий по комбинации флагов; кривая цикла по `PlatenDistance` (накопление
сэмплов между границами цикла); SPC/тренды по подушке; сглаживание аналога (скользящее
среднее/медиана с гистерезисом); hot-reload конфига. Каждое подключается как stateful-обработчик
над снимками, без переписывания ядра.

## 12. Открытые вопросы к железу (проверить при пусконаладке)

Реальный TCP-порт (`0x1076`), Unit ID (`0x00B1`, обычно 17), фактическое время отклика шлюза
при 0.5 с, соответствие DI↔фаза и полярность контактов, диапазоны аналогов для порогов. Всё
это — вход для тестера (Подзадача 2).
