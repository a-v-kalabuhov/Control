# Дизайн: методология пилотного деплоя PZP-03

**Дата:** 2026-07-20
**Статус:** черновик на ревью
**Ветка:** `feature/pzp-pilot-deploy`
**Задача:** PZP-03 — деплой пилота Control + коннектор USR-Modbus на ПК заказчика (ПЗП)

## Контекст

Пилотный запуск системы Wintime Control на площадке ПЗП: доказать заказчику
работоспособность на **одном ТПА** в течение **1–2 недель**. Если заказчик
удовлетворён — выделяет нормальное железо и подключает ещё ~20 ТПА.

**Условия площадки:**

- ПК заказчика: Windows 10 Pro, CPU Intel **i3-2100** (2 ядра / 4 потока, Sandy Bridge, ~2011),
  **6 ГБ ОЗУ**, Docker Desktop
  (автозапуск при входе в Windows уже включён), в общей сети завода.
- ТПА оснащён IoT-шлюзом **USR-IO424T-EWR**, в той же сети; связь ПК↔шлюз проверена,
  конфиг коннектора подготовлен на месте через Тестер (`config/<alias>.json`).
- Планшеты наладчиков — в той же сети; используются для **мобильного QR-флоу** (сверка
  пресс-формы при наладке камерой).
- **Нет штатного ИТ-специалиста.** Помощник может выехать на завод, но в командной
  строке работает тяжело. Деплой ведётся удалённо: помощник кликает и правит текстовые
  файлы, автор помогает по связи.

**Что снимает риск памяти:** базовый стек (api + postgres + mosquitto + эмулятор) уже
стабильно работает на VPS **1 ГБ / 1 ядро** без лагов. Значит на 6 ГБ связка
«api + postgres + mosquitto + коннектор, без эмулятора» — с большим запасом. Память
из главного риска исключается; `mem_limit` оставляем как страховку.

**Настоящее главное ограничение — не железо, а оператор.** Методология обязана быть
**turnkey**: минимум командной строки, максимум двойного клика и правки текстовых файлов.

## Цель

Дать помощнику **самодостаточный пакет-папку**, который он скачивает с облачного диска и
разворачивает на ПК заказчика в несколько двойных кликов (с удалённой помощью автора для
правки двух текстовых файлов и генерации сертификата), получая работающий пилот:
телеметрия одного ТПА через шлюз USR → Control → дашборд, видимый с других машин сети, и
рабочий мобильный QR-флоу на планшете наладчика.

## Область HTTPS (важно — мирит два требования)

Kestrel отдаёт **и HTTP, и HTTPS одновременно**:

- **Дашборды с других ПК цеха** → `http://<LAN-IP>:5000`. HTTPS для них не нужен, возни с
  сертификатами нет.
- **Планшеты наладчиков (QR)** → `https://<LAN-IP>`. Камера (`getUserMedia` в `html5-qrcode`,
  [QrScanner.vue](../../../Wintime-Control-Frontend/src/components/mobile/QrScanner.vue)) требует
  **secure-context** (HTTPS или `localhost`); по http планшет камеру блокирует. Поэтому только
  для планшетов поднимаем HTTPS с **локально-доверенным сертификатом mkcert**.

Следствие: корневой CA mkcert ставится **только на планшеты наладчиков** (их единицы), не на
все машины цеха. «HTTPS не нужен» остаётся верным для дашбордов; HTTPS точечно появляется там,
где без него камера не работает.

**Минимальная правка кода (нужна для http-дашбордов).** В Production API включает
`UseHttpsRedirection` (`Program.cs`), который редиректит http→https и ломает доступ к дашбордам
по http. Прячем редирект за конфиг-флаг (дефолт — прежнее поведение):

```csharp
if (!app.Environment.IsDevelopment() && builder.Configuration.GetValue("Https:Redirect", true))
    app.UseHttpsRedirection();
```

В пилотном `.env` — `Https__Redirect=false`. Другие среды не затронуты (флаг по умолчанию `true`).

## Не-цели (осознанно отложено до боевого деплоя)

| Отложено | Причина |
| --- | --- |
| HTTPS для дашбордов / публичный CA / reverse-proxy | Дашборды по http на LAN достаточно; HTTPS только для планшетов через mkcert. |
| Публикация образов в реестр (ghcr) | Требует поднять CI-публикацию ради одного деплоя. Доставка — тарболами (вариант B). |
| Режим коннектора `Source=api` | Обкатан локально на девмашине; для одного ТПА проще `file`. Переключим для 20 ТПА. |
| Windows-служба для коннектора/стека | Автозапуск обеспечивают Docker Desktop + `restart: unless-stopped`. |
| Бэкапы БД, мониторинг, алертинг | Пилот; данные не критичны. |
| Ужатие памяти как отдельная работа | Стек доказанно лёгкий (VPS 1 ГБ). |

## Ключевые решения

1. **Доставка образов — вариант B (`docker save`/`load`).** Образы собираются на машине
   автора, выкладываются на облачный диск тарболами, помощник качает и загружает на ПК.
   Сборка (тяжёлый `dotnet publish` + `npm build`) уходит с площадки; на ПК только рантайм.
   На ПК **не полагаемся на `docker pull`** — грузим **все 4 образа** тарболами (включая
   публичные postgres/mosquitto), чтобы деплой был офлайн-устойчивым.
2. **Артефакт — один пакет-папка.** Помощник работает только внутри неё; всё необходимое рядом.
3. **Turnkey через `.bat`.** Каждая команда спрятана в двойной-клик .bat; CLI руками — не трогаем.
4. **Секреты/параметры — в текстовых файлах** (`.env` + `config/<alias>.json`),
   задокументированных в `CONFIG_GUIDE.md`.
5. **Коннектор `Source=file`** — не зависит от старта API, без ApiKey и порядка запуска.
6. **Двойной доступ HTTP+HTTPS** (см. раздел выше): дашборды по http, планшеты по mkcert-https.
7. **Фиксированный LAN-IP ПК** (статика или DHCP-резерв) — сертификат mkcert выдаётся на IP;
   смена IP по DHCP сломала бы и сертификат, и закладки планшетов.

## Артефакт деплоя — структура пакета

```text
control-pzp-deploy/
├─ images/
│   ├─ api.tar            # .NET 9 + SPA
│   ├─ connector.tar      # Wintime.Connector.UsrModbus
│   ├─ postgres.tar       # postgres:16-alpine
│   └─ mosquitto.tar      # eclipse-mosquitto:2.0
├─ docker-compose.prod.yml # ПЗП-вариант: api + postgres + mosquitto + connector (без эмулятора)
├─ .env                    # все пароли/параметры (правится один раз)
├─ config/
│   ├─ machines.json       # список ТПА [{immId, immName, connectorAlias}] — immId вписывается на месте
│   └─ <alias>.json        # конфиг шлюза (готов Тестером; alias = connectorAlias из machines.json)
├─ certs/
│   ├─ pzp.pfx             # leaf-сертификат для LAN-IP (генерит 4-make-cert.bat)
│   └─ rootCA.pem          # корневой CA mkcert — ставится на планшеты
├─ mkcert.exe              # для генерации сертификата на месте (по фактическому IP)
├─ CAROOT/                 # корневой CA mkcert (rootCA.pem + rootCA-key.pem) автора
├─ mosquitto.conf
├─ 1-load-images.bat       # docker load всех тарболов
├─ 2-start.bat             # docker compose up -d
├─ 3-allow-lan.bat         # (от админа, один раз) firewall: входящие TCP 5000 и 443
├─ 4-make-cert.bat         # (автор удалённо) mkcert leaf-серт на фактический LAN-IP → certs/pzp.pfx
└─ CONFIG_GUIDE.md         # где что менять + установка rootCA на планшет
```

## `docker-compose.prod.yml` — ПЗП-вариант

Отличия от текущего `docker-compose.prod.yml` в репо:

- **Убрать сервис `emulator`.**
- **Добавить сервис `connector`** (образ USR-Modbus): volume `./config:/app/config:ro`,
  env `Connector__Source=file`, `Connector__SourcePath=config/machines.json`,
  `Connector__ConfigDir=config`, `Mqtt__BrokerHost=mosquitto`, `Mqtt__Port=1883`, `DOTNET_gcServer=0`.
- Все сервисы через `image:` (не `build:`) — образы уже загружены `docker load`.
- **api:** двойной листен `ASPNETCORE_URLS=http://+:8080;https://+:8443`; сертификат через env
  `ASPNETCORE_Kestrel__Certificates__Default__Path=/certs/pzp.pfx` +
  `ASPNETCORE_Kestrel__Certificates__Default__Password=${CERT_PASSWORD}`; volume `./certs:/certs:ro`;
  `DOTNET_gcServer=0`.
- Порты наружу: `5000:8080` (http, дашборды) и `443:8443` (https, планшеты). `1883` брокера
  наружу **не публикуем** — коннектор ходит к `mosquitto` внутри compose-сети.
- Каждому сервису — `restart: unless-stopped` (автоподъём после ребута) и `mem_limit`
  (страховка): postgres ~512m, api ~768m, connector ~256m, mosquitto ~64m.
- postgres: `shared_buffers=128MB`, `max_connections=20`, объём в volume.

Сеть: коннектор в контейнере обращается к шлюзу USR по LAN-IP — исходящий трафик из
контейнера в локальную сеть через NAT Docker Desktop работает штатно.

Миграции БД применяются **автоматически** при старте API (`db.Database.Migrate()`, Program.cs) —
ручной `dotnet ef database update` в деплое не нужен; на пустой БД схема создаётся сама.

## Секреты и параметры

**`.env`** (читается docker-compose автоматически):

| Ключ | Назначение |
| --- | --- |
| `DB_PASSWORD` | пароль PostgreSQL |
| `Bootstrap__AdminPassword` | пароль первого админа (создаётся при старте API) |
| `CERT_PASSWORD` | пароль PFX-сертификата (задаётся при генерации `4-make-cert.bat`) |
| `Https__Redirect` | `false` для пилота (иначе http-дашборды редиректятся на https) |
| `Connector__ApiKey` | не используется в `Source=file`; оставить пустым/убрать |

**`config/machines.json`** — список ТПА `[{immId, immName, connectorAlias}]`; на площадке вписать
`immId` (= `Imm.Id` из UI, см. ниже). **`config/<alias>.json`** — параметры шлюза
(IP/порт/UnitID/роли/калибровка); подготовлен Тестером, `<alias>` = `connectorAlias` из machines.json.

`CONFIG_GUIDE.md` построчно объясняет каждый параметр `.env` и конфига коннектора; как узнать
LAN-IP ПК (`ipconfig`); по какому адресу заходить с дашбордов (`http://<IP>:5000`) и с планшета
(`https://<IP>`); как установить `rootCA.pem` на планшет (Android/iOS — разово, с доверием).

## Связка `immId` — единственный тонкий шаг настройки

Коннектор публикует телеметрию в топик `control/imm/<immId>/telemetry`. `immId` **обязан
совпадать с `Imm.Id`** машины в БД Control, иначе дашборд её не покажет. `Imm.Id` — Guid,
генерится при создании ТПА в UI. Порядок:

1. Поднять Control → войти админом → создать Шаблон оборудования с `ConnectorType=usr-modbus`
   и ТПА в справочнике (с этим шаблоном).
2. Скопировать сгенерированный `Imm.Id` → вписать в поле `immId` записи в `config/machines.json`
   (делает автор удалённо).
3. Перезапустить контейнер коннектора (`docker compose restart connector` внутри .bat или
   через GUI Docker Desktop).

Это шаг, требующий удалённой руки автора; остальное помощник делает кликами.

## Как работает доверие сертификату (ликбез для площадки)

Ключевая мысль: **доверие — это локальная проверка на устройстве, а не сетевой запрос.**
Устройство никуда не ходит спрашивать CA. На каждом устройстве есть **хранилище доверенных
корней** (trust store). При подключении к `https://<IP>` сервер предъявляет свой сертификат,
браузер проверяет: «подписан ли он корнем, который **лежит у меня в хранилище**?». Да —
доверяет, нет — ругается. Сеть/домен для самой проверки не нужны.

- **Локальный CA (mkcert)** — это просто пара ключей + корневой сертификат, созданные с нуля
  одной командой. Приватным ключом корня подписываются серверные серты (на IP/имя). Любое
  устройство, у которого в хранилище лежит `rootCA.pem`, доверяет всему, что этот корень подписал.
- **«Установить сертификат на планшет»** = физически положить `rootCA.pem` в его trust store
  (Android — пара тапов в настройках). После этого `https://<IP>` для планшета — «зелёный замок».
- **Windows-домен / доменный CA (AD CS) — не используем:** его корень авто-раздаётся только
  машинам, введённым в домен; Android-планшеты в домен не входят → ставить всё равно руками, плюс
  зависимость от ИТ заказчика. Для 1–2 не-доменных планшетов mkcert проще и самодостаточен.
- **mkcert ничего не требует от инфраструктуры ПЗП** — ни домена, ни DNS, ни их ИТ. CA и пакет
  готовятся автором заранее; на площадке — выпуск серта на IP (одна команда) + установка корня
  на планшеты.

## Сертификат mkcert (для планшетов)

- Корневой CA mkcert автор генерит один раз и держит (`CAROOT/`); публичный `rootCA.pem`
  кладётся в пакет для установки на планшеты.
- **`4-make-cert.bat`** (автор запускает удалённо, узнав фактический LAN-IP ПК): `mkcert.exe`
  с `CAROOT` из пакета генерит leaf-сертификат на этот IP в формате PFX → `certs/pzp.pfx`, пароль
  пишется в `.env` (`CERT_PASSWORD`). Пересоздать при смене IP.
- Планшет: установить `rootCA.pem` в доверенные (Android: настройки → безопасность → сертификаты;
  iOS: профиль + включить полное доверие), затем открыть `https://<IP>` — камера/QR работают.

## LAN-доступ и брандмауэр

- **`3-allow-lan.bat` (запуск от админа, один раз)** добавляет два правила:
  входящие TCP `5000` (http, дашборды) и `443` (https, планшеты) через `netsh advfirewall`.
- MQTT `1883` в firewall **не открываем** — наружу не публикуется.

## Turnkey-скрипты (`.bat`)

- `1-load-images.bat` — `docker load -i images\*.tar` (все четыре).
- `2-start.bat` — `docker compose -f docker-compose.prod.yml up -d`.
- `3-allow-lan.bat` — netsh-правила на 5000 и 443 (требует прав админа; в шапке — подсказка
  «ПКМ → Запуск от имени администратора»).
- `4-make-cert.bat` — mkcert leaf-серт на фактический IP (запускает автор удалённо).

Автозапуск после ребута ПК обеспечивается автоматически: Docker Desktop стартует при входе в
Windows (уже включено) + `restart: unless-stopped` поднимает контейнеры. Отдельный автостарт
не нужен.

## Порядок ввода (адаптированная «лестница»)

Предполагается: связь ПК↔шлюз и конфиг коннектора уже проверены Тестером (сделано);
LAN-IP ПК зафиксирован (статика/DHCP-резерв).

1. Помощник: скачать пакет с облака, распаковать.
2. Двойной клик `1-load-images.bat` — загрузка образов.
3. Автор (удалённо): вписать пароли в `.env`; узнать LAN-IP; запустить `4-make-cert.bat`.
4. Двойной клик `2-start.bat` — подъём стека.
5. ПКМ→админ `3-allow-lan.bat` — открыть 5000 и 443 в firewall.
6. Браузер на ПК → `http://localhost:5000` → вход админом (пароль из `.env`).
7. Создать Шаблон (`usr-modbus`) + ТПА → получить `Imm.Id`.
8. Автор: вписать `immId` в `config/machines.json` → рестарт коннектора.
9. Дашборд с другой машины: `http://<LAN-IP>:5000`.
10. Планшет наладчика: установить `rootCA.pem` → открыть `https://<LAN-IP>` → проверить QR-скан.

## Критерии приёмки

1. Все 4 образа загружаются `docker load` и поднимаются `docker compose up -d` на ПК заказчика.
2. Дашборд открывается на ПК (`localhost:5000`) и с других машин сети (`http://<LAN-IP>:5000`);
   вход админом работает.
3. Пуск ТПА в автомат: `mode` переходит `idle → auto`, `cycleCounter` растёт на 1 за цикл впрыска;
   ТПА виден на дашборде «живым».
4. Планшет наладчика по `https://<LAN-IP>` запускает камеру и сканирует QR пресс-формы
   (мобильный QR-флоу работает).
5. После перезагрузки ПК стек и коннектор поднимаются сами (Docker Desktop autostart +
   `restart: unless-stopped`).
6. Эмулятор в проде не запускается.

## Риски и открытые вопросы

1. **Установка CA на планшеты.** Разовый ручной шаг на каждом планшете (Android проще, iOS —
   профиль + полное доверие). Митигация — планшетов единицы; инструкция в `CONFIG_GUIDE.md`,
   автор помогает удалённо.
2. **Стабильность LAN-IP.** Сертификат и закладки привязаны к IP; смена по DHCP ломает HTTPS.
   Митигация — статика/DHCP-резерв как предусловие; при смене — перегенерить `4-make-cert.bat`.
3. **Firewall от админа.** `3-allow-lan.bat` требует прав администратора; если у помощника
   учётка без прав — понадобится админ-доступ. Уточнить заранее.
4. **Совпадение `immId`.** Ошибка = «пустой» дашборд при работающей телеметрии. Митигация —
   шаг делает автор удалённо, сверяя Guid; проверка по приходу телеметрии в топик.
5. **Приватный ключ CA на ПК.** `CAROOT/` с ключом кладётся на ПК ради генерации серта на месте.
   Это одноразовый локальный CA пилота (не публичный) — приемлемо; после пилота CA утилизируется.
6. **Короткий импульс Injection.** Если сигнал впрыска короче периода опроса — цикл проскочит
   (TESTING_ONSITE, ступень 1.3). На связке проверено Тестером; при сбое `cycleCounter` —
   уменьшить `pollIntervalMs`.

## Дальнейшее (боевой деплой, после успеха пилота)

Отдельный чек-лист, когда заказчик выделит железо и ~20 ТПА:

- Публикация образов в реестр (ghcr) + `docker compose pull` (вариант C).
- Коннектор `Source=api` (список ТПА из `GET /api/connectors/usr-modbus/machines`).
- Полноценный HTTPS (доменное имя + доверенный сертификат) — снимет установку CA на каждый
  планшет и даст https дашбордам.
- Устойчивость: бэкапы БД, ретенция логов, при необходимости — Windows-служба.
- Пересмотр `mem_limit`/ресурсов под реальное железо и число ТПА.
