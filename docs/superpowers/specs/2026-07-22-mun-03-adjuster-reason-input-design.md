# MUN-03 — ввод причины простоя наладчиком (выравнивание с замыслом)

> Дата: 2026-07-22 · Задача: MUN-03 «Журнал простоев» (MVP Мун)

## Контекст и проблема

Замысел учёта простоев:

- Журнал простоев заполняется **автоматически** (воркер `DowntimeDetectionWorker`):
  по эффективному статусу ТПА и наличию активного задания создаётся/закрывается событие
  «простой», длительность считается сама.
- Причину простоя приложение определить не может — это работа человека.
- **Первичный** ввод причины — наладчик в своём (мобильном) интерфейсе.
- **Дублирующий** путь — менеджер в модуле «Журнал простоев» (десктоп) может изменить
  причину любой записи.

Расхождение реализации с замыслом:

- Бэкенд `PATCH /api/downtime/events/{id}` уже позволяет менять причину и **разрешён роли
  `Adjuster`** — то есть первичный путь на сервере готов.
- Десктопный журнал (менеджер) правку причины умеет — дублирующий путь работает.
- **Мобильный интерфейс наладчика правку причины НЕ умеет.** Наладчик может лишь вручную
  создать простой (сразу с причиной) и завершить активный. У существующего события
  (в частности авто-простоя с «Причина не указана») **нет действия «указать/изменить
  причину»**. Первичный путь ввода причины в UI отсутствует.

## Границы (согласовано)

В скоупе — **только UI ввода причины наладчиком**. Вне скоупа (отложено в BL-21):
комментарий к простою, поля «автор изменения» и «время изменения» причины/комментария,
любые миграции БД. Бэкенд, API-слой фронта и Pinia-store не меняются.

## Решение

Изменение точечное, целиком в одном файле:
`Wintime-Control-Frontend/src/views/mobile/MobileDowntimeView.vue`.

1. **Действие на карточке события** (под блоком причины; и для активных, и для
   завершённых простоев — как в десктопном журнале, где причину активного тоже можно
   править):
   - причина не указана → кнопка **«Указать причину»** (акцентная, `type="primary"`);
   - причина указана → кнопка **«Изменить причину»** (ненавязчивая, `text`/`plain`).
   - Применимо к авто-простоям (`isAuto=true`) и к ручным.

2. **Отдельный модальный диалог «Причина простоя»**:
   - селект причин с цветными тегами «Плановый»/«Аварийный» — переиспользуется разметка
     из существующего диалога регистрации простоя;
   - предзаполнен текущим `reasonId` события;
   - кнопки «Отмена» / «Сохранить»; при попытке сохранить без причины —
     `ElMessage.warning('Выберите причину')`, запрос не уходит.

3. **Сохранение**: `downtimeApi.updateEvent(event.id, { reasonId })`
   (`PATCH /downtime/events/{id}`). Передаём только `reasonId` — `comment` и `endTime`
   не трогаем (PATCH оставляет их без изменений). При успехе — тост «Причина обновлена»,
   закрыть диалог, `loadEvents()`.

## Поток данных

```
Наладчик → карточка события → «Указать/Изменить причину»
  → диалог (селект активных причин из mobileStore.downtimeReasons)
  → Сохранить → downtimeApi.updateEvent(id, { reasonId })
  → PATCH /api/downtime/events/{id} (роль Adjuster)  [без изменений]
  → тост «Причина обновлена» → loadEvents()
```

## Что НЕ делаем (YAGNI / отложено)

- Поле и логика комментария (BL-21).
- Поля «автор изменения причины» и «время изменения» + их отображение менеджеру (BL-21).
- Миграции БД, изменения модели `Event`, контроллера, `downtime.js`, `mobile.js`.
- Изменения десктопного журнала.
