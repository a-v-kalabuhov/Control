# ADR-0005: COV-фильтрация телеметрии — Вариант B (удержание последнего значения)

- **Статус:** Accepted
- **Дата:** 2026-06-16

## Контекст

ТПА шлют телеметрию по таймеру (раз в N секунд), а не по изменению. Большинство значений
между соседними сообщениями либо не меняются, либо дребезжат в пределах инструментальной
погрешности датчика (например, температура зоны ±0.1 °C при пороге значимости 0.5 °C).
Если писать в `Telemetry` каждое сообщение как есть, получаем:

1. шумовой ряд, в котором незначимый дребезг неотличим от реальных микро-изменений;
2. плохое сжатие на стороне хранилища — соседние значения почти всегда чуть-чуть разные.

При этом терять строки нельзя: отчёты и графики строятся по равномерной временной сетке,
и пропуск строки = «дыра» во времени, которую потом надо домысливать при чтении.

В шаблоне датчика (`SensorTemplate`) уже есть поле `Threshold` — порог значимого изменения,
задаваемый инженером при настройке оборудования. COV-фильтрация (Change of Value) применяется
в `ValidateTelemetryDataHandler` после валидации типов.

Возник вопрос: **что именно писать в БД, когда изменение датчика не превысило порог.**

## Решение

Применяем **Вариант B — удержание последнего зафиксированного значения (deadband / hold-last)**.

Для каждого датчика с `Threshold > 0`:

- если изменение относительно кешированного значения **в пределах порога** — в строку БД
  пишется **кешированное (предыдущее) значение**, а не присланное; кеш не обновляется;
- если изменение **превысило порог** (или датчик встретился впервые) — пишется новое
  значение, кеш обновляется им.

Ключевое: **строка пишется всегда**, на каждое сообщение. Меняется только *значение* поля,
а не факт наличия строки. Временная сетка остаётся равномерной без дыр.

Граничные случаи проходят **без фильтрации** (значение пишется как пришло):

- `Threshold == 0` — порог не задан, фильтр для датчика выключен;
- первое сообщение от ТПА (кеш пуст);
- сообщение с нарушенным порядком (out-of-order, `timestamp` раньше кеша);
- первое сообщение после периода offline;
- типы `string` и `boolean` — любое несовпадение считается значимым (порог к ним не применяется).

## Альтернативы

- **Вариант A — drop-unchanged (не писать строку, если изменение в пределах порога).**
  Отвергнуто: ломает равномерную временную сетку. Отчёты и графики пришлось бы дополнять
  логикой «значение действует до следующей строки», а пропуск становится неотличим от
  реального отсутствия связи (offline). Меньше строк ценой усложнения каждого читателя данных.

- **Вариант C — писать всегда ровно присланное значение (COV вообще не применять).**
  Отвергнуто: в БД остаётся инструментальный шум; ряд хуже сжимается; незначимый дребезг
  неотличим от значимого микро-изменения. Порог `Threshold` из шаблона тогда не имеет смысла.

- **Вариант B′ — писать строку всегда, но при изменении в пределах порога ставить пометку
  «не изменилось» (отдельный флаг), сохраняя присланное значение.** Отвергнуто как
  избыточное для MVP: добавляет колонку и логику в каждый запрос ради восстановления
  под-порогового шума, который мы как раз и хотим убрать.

## Последствия

- **Плюсы:** равномерная сетка без дыр; шум в пределах порога подавлен; повторяющиеся
  значения отлично сжимаются (в т.ч. при будущем переходе на TimescaleDB compression);
  порог настраивается per-sensor через шаблон.
- **Компромисс / технический долг:** в БД пишется не дословно присланное значение, а
  удержанное предыдущее, когда разница ≤ `Threshold`. При отладке «датчик показывал X,
  в отчёте Y» расхождение всегда ограничено порогом (инженерно объявленным незначимым),
  но об этом нужно помнить. Точное сырое значение в этом случае не сохраняется нигде.
- **Граница применимости:** фильтр держит состояние в in-memory кеше (`IImmCache`).
  После рестарта/offline первое сообщение всегда пропускается без фильтрации — это
  осознанно, чтобы не «удерживать» значение через разрыв.
- Если для какого-то сценария понадобится дословное сырьё — отключается точечно
  установкой `Threshold = 0` в шаблоне датчика, без изменения кода.
