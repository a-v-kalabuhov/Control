# ADR-0002: Конечный автомат задания в сущности ShiftTask

- **Статус:** Accepted
- **Дата:** 2026-06-15

## Контекст

Жизненный цикл задания `Draft → Issued → Setup → InProgress → Completed → Closed`
(плюс `Setup → Issued` при отмене наладки) был размазан по 7 экшенам `TasksController`
в виде `if (task.Status != X) return BadRequest(...)` с ручной мутацией статуса и меток
времени. Сущность `ShiftTask` была анемичной. Добавить переход или статус означало
искать и править логику по контроллерам — высокий риск рассинхронизации.

## Решение

Переходы — это **методы сущности** `ShiftTask`: `Issue()`, `StartSetup()`,
`CompleteSetup()`, `CancelSetup()`, `VerifyMold()`, `Complete()`, `Close()`. Каждый
проверяет исходный статус через приватный `EnsureStatus()` и кидает `DomainException`
при недопустимом переходе (см. [ADR-0003](0003-domain-exception-to-http-400.md)).

Побочные эффекты, не относящиеся к состоянию задания (перевод режима ТПА через
`IEmulatorControlService`), **остаются в контроллере** — сущность не зависит от инфраструктуры.

## Альтернативы

- **Оставить в контроллере** — отвергнуто: дублирование, размазанный инвариант.
- **Отдельный доменный сервис `TaskStateService`** — отвергнуто: для 6 статусов лишний
  слой; логика органична для самой сущности.
- **Паттерн State (класс на каждый статус)** — отвергнуто: оверинжиниринг для линейного
  автомата с одним ответвлением.

## Последствия

- Переходы и их гарды — в одном месте, покрыты unit-тестами без поднятия HTTP/БД.
- Контроллер сведён к: загрузить → вызвать метод → сохранить → (побочка эмулятора).
- `Core` получает тип `DomainException` (внутренняя зависимость Core, допустимо).
- Прецедент: дальнейшую доменную логику складываем в сущности, а не в контроллеры.
