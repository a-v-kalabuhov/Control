# ADR-0003: DomainException → HTTP 400 через IExceptionHandler

- **Статус:** Accepted
- **Дата:** 2026-06-15

## Контекст

Перенос инвариантов в доменные методы (см. [ADR-0002](0002-task-state-machine-in-entity.md))
требует механизма: как нарушение доменного правила превращается в HTTP-ответ, не возвращая
`try/catch` и `if/return BadRequest` обратно в контроллеры.

Дополнительно вскрылся дефект контракта: фронтенд читает `error.response.data.message`,
а старый `return BadRequest("строка")` отдавал голую JSON-строку без поля `message` —
конкретный текст ошибки до пользователя **не доходил** (всегда фолбэк «Ошибка …»).

## Решение

Доменные нарушения кидают `DomainException` (в `Wintime.Control.Core.Exceptions`).
Глобальный `DomainExceptionHandler : IExceptionHandler` маппит его в **HTTP 400** с телом
`{ "message": "..." }`. Регистрация: `AddExceptionHandler<>()` + `AddProblemDetails()` +
`app.UseExceptionHandler()` в начале конвейера. Прочие исключения handler пропускает дальше.

## Альтернативы

- **`try/catch` в каждом экшене** — отвергнуто: тот же размазанный шаблонный код.
- **Тело `ProblemDetails`** (`{ detail, status, ... }`) — отвергнуто: фронт читает
  `data.message`; смена формы тела сломала бы отображение ошибок.
- **Тип-результат `Result<T>`** вместо исключений — отвергнуто: крупный рефакторинг
  ради того же эффекта; исключения здесь — действительно исключительный путь.

## Последствия

- Контроллеры чисты; любой будущий `DomainException` из любого слоя автоматически → 400.
- Контракт ошибки `{ message }` совпадает с success-ответами (`Ok(new { message })`).
- Побочный фикс: специфичный текст ошибки теперь реально показывается пользователю.
- Тело ошибки для доменных нарушений — **не** `ProblemDetails`; если в будущем перейдём
  на `ProblemDetails` повсеместно, нужно синхронно поправить фронтенд (новый ADR).
