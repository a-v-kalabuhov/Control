# ADR-0004: Единственный источник роли — User.Role; Identity-роли не используются

- **Статус:** Accepted
- **Дата:** 2026-06-16

## Контекст

Роль пользователя хранилась в двух местах одновременно:

1. **`User.Role`** (enum `Core.Enums.UserRole`) — колонка в таблице `Users`.
2. **Identity-роли** — таблицы `AspNetRoles` / `AspNetUserRoles`, наполнялись через
   `userManager.AddToRoleAsync` (в сидировании `Program.cs` и в `PersonnelController`).

При этом авторизация на каждом запросе фактически опиралась **только на `User.Role`**:
JWT-клейм `"role"` строится из `user.Role.ToString()` ([JwtTokenService]), а
`RoleClaimType = "role"` ([Program.cs]) — значит `[Authorize(Roles=…)]` проверяет именно
этот клейм. Identity-роли писались, но в авторизации **не читались никогда**. Так же
`User.Role` напрямую участвует в EF-запросах отчётов
(`ReportService.Where(u => u.Role == UserRole.Adjuster)`).

Это два источника правды, которые могут молча разойтись: смена роли через `User.Role`
без синхронизации `AspNetUserRoles` (или наоборот) ничего не сломает явно, но создаёт
«мёртвый» источник, выглядящий как настоящий — будущий код, меняющий роль через
`RoleManager`/`AddToRoleAsync`, увидит изменение в БД, а доступы не поменяются.

## Решение

Единственный источник правды по роли — **`User.Role` (enum)**. Одна роль на пользователя.

- Прекращаем любую запись Identity-ролей: убраны `AddToRoleAsync` /
  `RemoveFromRolesAsync` из `PersonnelController` и сидирование `AspNetRoles` в `Program.cs`.
- JWT-клейм `"role"`, `ReportService` и `[Authorize(Roles=…)]` остаются как есть —
  контракт авторизации **не меняется**.
- Первый администратор для production создаётся при старте из секрета
  (`Bootstrap:AdminPassword`, через env `Bootstrap__AdminPassword`), идемпотентно и
  только если ни одного админа ещё нет. Пароль никогда не хранится в исходниках.
- Демо-аккаунты по-прежнему создаются только при `DemoMode=true`.

Таблицы `AspNetRoles` / `AspNetUserRoles` остаются в схеме (инертны) — миграция БД не
требуется. Регистрация Identity (`AddIdentity<User, IdentityRole>`) сохранена.

## Альтернативы

- **Identity-роли ведущие** (удалить `User.Role`, JWT из `GetRolesAsync`, отчёты через
  join к `AspNetUserRoles`) — отвергнуто: требует миграции БД с удалением колонки и
  переносом данных, переписывания EF-запросов отчётов через Identity-таблицы и доп.
  обращения к БД при выпуске токена. Больший объём и риск для уже развёрнутого ПЗП —
  ради канонической формы Identity, тогда как многоролевость бизнесом не требуется.
- **Оставить оба источника, синхронизировать вручную** — отвергнуто: это и есть текущая
  ловушка; синхронизация рано или поздно разъедется.

## Последствия

- Один источник роли; расхождение клейма JWT и политик доступа невозможно по построению.
- Закрыта дыра с захардкоженными паролями: production-админ — из секрета окружения.
- Контракт авторизации и DTO персонала не изменились → фронтенд и развёрнутые БД не ломаются.
- Компромисс: одна роль на пользователя (многоролевость Identity недоступна). Если
  понадобится несколько ролей — это новый ADR с переходом на Identity-роли как источник.
- `AspNetRoles`/`AspNetUserRoles` остаются пустыми инертными таблицами; их физическое
  удаление — отдельная необязательная миграция, при желании в будущем.
