# ADR-0001: Снапшот гнёздности в ImmCycle.Cavities

- **Статус:** Accepted
- **Дата:** 2026-06-15

## Контекст

Выработка задания (`ActualQuantity`) и расход сырья считаются по гнёздности ПФ
(`Mold.Cavities`). Но `Mold.Cavities` — **изменяемое** поле: при ремонте гнёзда
заглушают. Если считать историю по текущему значению, завершённые циклы получают
пересчёт задним числом — выпуск «плывёт» после ремонта формы.

Ранее выработка действительно бралась из живого `Mold.Cavities` (в обработчике циклов
и в отчёте), что нарушало доменное правило, зафиксированное в `CLAUDE.md`.

## Решение

Добавляем `ImmCycle.Cavities` (int) — **снапшот гнёздности на момент записи цикла**.
Заполняется из `Mold.Cavities` при создании цикла. Вся выработка и расход сырья
считаются по сумме снапшотов. Для старых записей (`Cavities = 0`) — fallback на текущее
`Mold.Cavities`. Реализовано: задача MUN-08.

## Альтернативы

- **Считать по живому `Mold.Cavities`** — отвергнуто: ровно та ошибка пересчёта истории.
- **Сразу `ImmCycle.MoldVersionId` → сущность `MoldVersion`** (полная версия конфигурации
  ПФ с весами и статусом) — отложено до РОСОМС. Для MVP избыточно; станет фундаментом
  журнала ремонтов. Этот ADR будет заменён соответствующим, когда `MoldVersion` появится.

## Последствия

- Историческая выработка стабильна при изменении гнёздности.
- Веса (`PartWeightGrams`, `RunnerWeightGrams`) **не** снапшотятся в MVP — расход сырья
  по ним всё ещё берёт текущие значения ПФ. Принятый компромисс MVP; снимется с `MoldVersion`.
- Миграция `AddImmCycleCavities` (`integer NOT NULL DEFAULT 0`); ноль = маркер legacy-записей.
- Правило также записано в `CLAUDE.md` (Domain Rules → Гнёздность).
