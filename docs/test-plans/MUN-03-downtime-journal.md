# План ручного тестирования — MUN-03 «Журнал простоев»

## Предусловия

| Что | Значение |
|---|---|
| Окружение | API запущен (`dotnet run --project Wintime.Control.API`), фронт собран/`npm run dev` |
| Пользователи | Наладчик (`Adjuster`), Начальник цеха (`Manager`), Админ (`Admin`) |
| Данные | ≥2 активных ТПА, ≥3 причины простоя в справочнике (минимум одна `Planned` и одна `Emergency`) |
| Браузеры | Десктоп (журнал, справочник) + планшет/мобильный режим (регистрация простоя) |

---

## Блок 1. Мобильный интерфейс наладчика — регистрация простоя

| # | Шаги | Ожидаемый результат |
|---|---|---|
| 1.1 | Войти как `Adjuster`, открыть вкладку «Простои» | Список простоев за сегодня; кнопка «Зарегистрировать простой» |
| 1.2 | Нажать «Зарегистрировать простой», выбрать ТПА и причину, «Начать простой» | Карточка появилась, статус «Активный» (оранжевая), время начала проставлено, «В процессе…» |
| 1.3 | Попытаться начать без выбора ТПА | Предупреждение «Выберите машину», запрос не уходит |
| 1.4 | Попытаться начать без выбора причины | Предупреждение «Выберите причину» |
| 1.5 | У активного простоя нажать «Завершить простой» | Статус → «Завершён» (серая карточка), показаны время конца и длительность |
| 1.6 | Зарегистрировать простой, обновить страницу (F5) | Активный простой сохранился (данные из БД, не из памяти) |
| 1.7 | Проверить, что в выпадающих списках причин есть цветные теги «Плановый»/«Аварийный» | Теги отображаются корректно по `type` |

## Блок 2. Десктопный журнал (Начальник цеха / Админ)

| # | Шаги | Ожидаемый результат |
|---|---|---|
| 2.1 | Войти как `Manager`, открыть «Журнал простоев» | Таблица: ТПА, Начало, Конец, Длительность, Причина; по умолчанию период = сегодня |
| 2.2 | Проверить простои, созданные в Блоке 1 | Отображаются; активный — тег «Активный», длительность «—» |
| 2.3 | Фильтр по периоду: выбрать диапазон, охватывающий записи, «Найти» | Показаны записи из диапазона |
| 2.4 | Фильтр по периоду без записей | «Простоев не найдено» |
| 2.5 | Фильтр по конкретному ТПА | Только простои выбранного ТПА |
| 2.6 | Сбросить фильтр ТПА (clearable) → «Найти» | Все машины снова в выборке |
| 2.7 | «Изменить причину» у записи → выбрать другую причину → «Сохранить» | Тост «Причина обновлена», в таблице новая причина без перезагрузки |
| 2.8 | «Изменить причину», нажать «Сохранить» без выбора причины | Предупреждение «Выберите причину» |
| 2.9 | Проверить счётчик «Записей: N» внизу | Совпадает с числом строк |
| 2.10 | Граница суток: создать простой, убедиться что попадает в фильтр `to = дата+23:59:59` | Запись за текущий день не теряется на границе |

## Блок 3. Справочник причин простоя — CRUD

> Доступ: `Admin`/`Manager`. Открыть «Справочники → Причины простоев».

### 3.1 Создание
| # | Шаги | Ожидаемый результат |
|---|---|---|
| 3.1.1 | «Добавить причину», заполнить наименование + тип, «Сохранить» | Тост «Причина создана», строка в таблице |
| 3.1.2 | Сохранить с пустым наименованием | Валидация формы «Введите наименование», запрос не уходит |

### 3.2 Редактирование
| # | Шаги | Ожидаемый результат |
|---|---|---|
| 3.2.1 | «Редактировать» → сменить имя/тип/статус → «Сохранить» | Тост «Причина обновлена», изменения в таблице |
| 3.2.2 | Переименовать в имя **уже существующей** причины (проверить и в другом регистре, с пробелами) | Ошибка **«Причина с таким наименованием уже существует»** (409), изменения не сохранены |
| 3.2.3 | Сохранить причину, не меняя имя (только тип/статус) | Успех — собственное имя не считается дубликатом |
| 3.2.4 | Очистить наименование и сохранить | Ошибка «Наименование причины обязательно» (форма блокирует пустое; пробелы → 400 с сервера) |
| 3.2.5 | Установить статус «Не активна», сохранить | Причина остаётся в **справочнике** с тегом «Не активна»… |
| 3.2.6 | …но открыть форму простоя в мобильном/журнале | …и **отсутствует** в списках выбора причин (грузятся только активные) |

### 3.3 Удаление
| # | Шаги | Ожидаемый результат |
|---|---|---|
| 3.3.1 | Создать новую причину, **не** использовать её → «Удалить» → подтвердить | Тост «Причина удалена», строка исчезла (физическое удаление) |
| 3.3.2 | Попытаться удалить причину, **использованную** в журнале простоев | Ошибка **«Причина использовалась в журнале простоев и не может быть удалена. Сделайте её неактивной.»** (409), причина остаётся |
| 3.3.3 | В диалоге подтверждения нажать «Отмена» | Ничего не удаляется, ошибок нет |
| 3.3.4 | После 3.3.2 заархивировать ту же причину (статус «Не активна») | Архивирование как альтернатива удалению работает |

## Блок 4. Контроль доступа (негативные сценарии)

| # | Шаги | Ожидаемый результат |
|---|---|---|
| 4.1 | Под `Adjuster` нет пункта меню «Журнал простоев» / «Причины простоев» | Пункты скрыты (роль `Admin/Manager`) |
| 4.2 | Под `Adjuster` обратиться к `PUT /api/downtime/reasons/{id}` напрямую (DevTools/Postman) | `403 Forbidden` |
| 4.3 | Под `Adjuster` обратиться к `DELETE /api/downtime/reasons/{id}` | `403 Forbidden` |
| 4.4 | Без токена обратиться к любому эндпоинту `/api/downtime/*` | `401 Unauthorized` |
| 4.5 | Под `Admin` повторить ключевые операции Блока 3 | Доступ есть (Admin ≥ Manager) |

---

## Блок 5. Граничные/прочие проверки

| # | Шаги | Ожидаемый результат |
|---|---|---|
| 5.1 | Завершить простой, у которого нет активной записи (двойной «Завершить») | Сервер `404 «Активный простой не найден»`, UI не падает |
| 5.2 | Длительность: простой ~1 ч 5 мин | Формат «1 ч 5 мин»; <1 ч → «N мин» |
| 5.3 | Часовые пояса: время начала/конца в UI совпадает с локальным | Корректное отображение (UTC в БД, локаль на клиенте) |
| 5.4 | Активный простой → «Изменить причину» в журнале | Причину можно поменять и для незавершённого простоя |

---

## Критерии приёмки
- Блоки 1–4 пройдены без дефектов.
- Подтверждены два ключевых правила нового функционала: **дубликат имени запрещён** (3.2.2) и **удаление используемой причины запрещено** (3.3.2).
- Контроль доступа: наладчик не может править/удалять причины (4.2–4.3).
