# UsrIoPoller — описание кода и контекст для доработки

Документ описывает назначение, архитектуру и инварианты проекта `UsrIoPoller` — read-only
клиента опроса модуля ввода-вывода **USR-IO424T-EWR** по Modbus TCP. Предназначен для
передачи вместе с исходниками в Claude Code, чтобы довести проект до полноценного
приложения с интерфейсом и дополнительными функциями.

---

## 1. Назначение и контекст

Проект опрашивает сетевой модуль USR-IO424T-EWR и декодирует его регистры в осмысленные
значения тегов. Прикладная задача — съём сигналов с термопластавтомата (ТПА) **BORCHE** с
контроллером **KEBA**: контроллер релейными (беспотенциальными) выходами заводит сигналы на
сухие контакты USR, USR отдаёт их по Modbus.

Снимается **6 тегов**:

| Тег | Смысл | Тип |
|---|---|---|
| MoldClose (DI1) | закрытие формы | дискретный 0/1 |
| Injection (DI2) | впрыск | дискретный 0/1 |
| Ejection (DI3) | выталкивание | дискретный 0/1 |
| MoldOpen (DI4) | открытие формы | дискретный 0/1 |
| PlatenDistance (AI1) | положение подвижной формы (кривая цикла) | аналог 0–10 В |
| Cushion (AI2) | подушка — показатель стабильности цикла | аналог 0–10 В |

Требуемая частота опроса — **0,5 с**. Программа **только читает** данные; запись в устройство
намеренно не реализуется.

---

## 2. Специфика устройства (важно для любой доработки)

USR-IO424T-EWR реализует модель данных Modbus. Ключевые факты, которые нельзя терять при
расширении:

- **Регистры читаются разными функциональными кодами** в зависимости от типа данных:
  дискретные входы — FC 0x02 (discrete inputs), аналоговые/счётчики — FC 0x03/0x04
  (holding/input registers), обратное чтение реле — FC 0x01 (coils). Поэтому у каждого тега
  в конфиге есть поле `access`, определяющее код чтения. Это не косметика: биты и слова
  приходят в **разном формате ответа** (упакованные биты против массива 16-битных слов).
- **Порядок байт в регистре — big-endian**; NModbus снимает его с провода, значения приходят
  уже в host-порядке (отдельный свап не нужен для однорегистровых значений).
- **Аналоговые значения — в милливольтах напрямую** (`0x1000` = 4096 = 4.096 В). Пересчёт в
  вольты — линейный: `scale=0.001`.
- **Modbus-адрес устройства (Unit ID)** обычно 17; TCP-порт по стандарту 502, но у USR он
  конфигурируемый (регистр `0x1076`) — **уточнять на реальном устройстве**.
- **Частота 0,5 с — комфортная**: время отклика устройства в документации не указано,
  но при таком периоде запас огромный. Для быстрого опроса (десятки мс) отклик надо измерять
  на железе — это ограничение устройства, не кода.

Полная карта регистров и нюансы — в отдельной спецификации `USR-IO424T-EWR_Modbus_Read_Spec.md`
(если она передаётся вместе).

## 3. Архитектура

Поток данных однонаправленный, с жёсткой границей между преобразованием без состояния и
логикой с состоянием:

```
registers.json
      │  ConfigLoader (парсинг + валидация)
      ▼
PollingConfig ──► BatchPlanner ──► ReadBlock[]        (группировка в минимум запросов)
      │
      ▼
Poller (PeriodicTimer 0.5s)
      │  на каждый тик:
      │   1. IModbusReader читает каждый ReadBlock         ← NModbus (TCP)
      │   2. TagDecoder декодирует regs → TagSample        ← STATELESS
      │   3. складывает в TagSnapshot
      ▼
TagSnapshot ──► SnapshotReady (event) ──► UI / лог / БД
      │
      └────────► ITagProcessor[]                           ← STATEFUL
                    ├─ EdgeCycleProcessor   (детекция цикла)
                    └─ CushionStabilityProcessor (качество)
```

**Главный инвариант — разнесение stateless / stateful:**

- **Stateless** (конфиг + декодер): превращает сырой регистр в мгновенное значение тега.
  Формула `scale/offset` для аналога либо таблица `valueMap` для дискретных. Работает
  над одним сэмплом, состояния не хранит.
- **Stateful** (процессоры за `ITagProcessor`): дельты счётчиков, детекция завершённого
  цикла с мёртвым временем, усреднение шума аналога, привязка замера к фазе цикла. Работает
  над потоком снимков, хранит состояние.

Эту границу нельзя стирать: конфиг не должен обрастать выражениями/логикой, а stateful-логика
не должна лезть в декодер. Всё, что требует памяти между опросами, — это `ITagProcessor`.

## 4. Файлы и ключевые типы

| Файл | Содержимое |
|---|---|
| `Config.cs` | Модель конфига: enum `ModbusAccess`/`RawType`/`TargetKind`; records `DeviceConfig`, `RegisterDef`, `PollingConfig`. |
| `ConfigLoader.cs` | Загрузка/валидация JSON: парсинг hex-адресов, резолв `valueMap` в целевой тип, DTO-слой, `InvalidConfigException`. |
| `ModbusReader.cs` | Интерфейс `IModbusReader` (4 читающих операции) + адаптер `NModbusReader` (Modbus TCP). |
| `Pipeline.cs` | `TagSample`, `ReadBlock`, `BatchPlanner` (группировка запросов), `TagDecoder` (stateless-декод). |
| `Poller.cs` | `TagSnapshot`, интерфейс `ITagProcessor`, класс `Poller` (таймер, чтение, декод, раздача, переподключение). |
| `Processors.cs` | Примеры stateful-процессоров: `EdgeCycleProcessor`, `CushionStabilityProcessor`. |
| `registers.json` | Конфигурация под 6 тегов ТПА. |
| `Program.cs` | Точка входа: сборка конфига, опросчика и процессоров. |
| `UsrIoPoller.csproj` | .NET 9, зависимость NModbus `3.0.*`. |

**Ключевые типы:**

- `RegisterDef` — описание тега. Две взаимоисключающие ветки: `Scale`/`Offset` (аналог,
  `Target=Double`) либо `ValueMap` (дискретный, `Target` = Bool/Int/String). Свойства
  `IsBitAccess` и `IsMapped` управляют веткой декодирования.
- `TagSample` — результат декодирования: `RawNumeric` (сырое число), `Value` (целевое
  значение, boxed `object?`), `Physical` (double для аналога), `Unit`, `TimestampUtc`.
- `TagSnapshot` — снимок всех тегов за один опрос.
- `Poller` — оркестратор. Событие `SnapshotReady` и метод `AddProcessor` — основные точки
  подключения потребителей. Метка времени снимается **после** ответа устройства.
- `IModbusReader` — абстракция чтения; текущая реализация на NModbus, но заменяема
  (например, на FluentModbus или RTU-транспорт).

## 5. Схема конфигурации (`registers.json`)

**Секция `device`:**

| Поле | Тип | Дефолт | Смысл |
|---|---|---|---|
| `host` | string | — | IP устройства (обязателен) |
| `port` | int | 502 | TCP-порт (уточнить, см. §2) |
| `unitId` | byte | 17 | Modbus-адрес устройства |
| `pollIntervalMs` | int | 500 | период опроса |
| `timeoutMs` | int | 1000 | таймаут чтения |
| `maxBatchGap` | ushort | 0 | макс. зазор адресов для склейки в один запрос |

**Каждый элемент `registers`:**

| Поле | Тип | Смысл |
|---|---|---|
| `name` | string | уникальное имя тега |
| `address` | string | адрес: `"0x0020"` или десятичный |
| `access` | enum | `coil` / `discreteInput` / `holdingRegister` / `inputRegister` |
| `rawType` | enum | `bit` / `uint16` / `int16` |
| `count` | ushort | число регистров (обычно 1) |
| `scale`, `offset` | double | линейный пересчёт для аналога: `физ = raw*scale + offset` |
| `unit` | string | единица (например, `"V"`) |
| `target` | enum | `bool` / `int` / `string` / `double` |
| `valueMap` | object | таблица `"raw": целевое`, напр. `{"1": true, "0": false}` |

Разбор enum — без учёта регистра. Если задан `valueMap` — применяется он (дискретная ветка),
`scale/offset` игнорируются. Один и тот же `raw=1` может мапиться в `true`, `"ON"` или `100` —
определяется парой `target` + `valueMap`.

## 6. Инварианты, которые нельзя нарушать при доработке

1. **Read-only.** Никаких FC записи (0x05/0x06/0x0F/0x10). Программа не меняет состояние
   устройства. Если понадобится управление — это отдельное осознанное решение, не «между делом».
2. **Граница stateless/stateful** (см. §3). Логика с памятью — только в `ITagProcessor`.
3. **Батчинг.** Смежные регистры одного `access` читаются одним запросом. Не возвращаться к
   схеме «один тег — один round-trip».
4. **Метка времени — после ответа устройства**, не после отправки запроса (важно для
   привязки замеров к фазе и для честности временных рядов).
5. **Абстракция транспорта.** Прикладной код зависит от `IModbusReader`, а не от NModbus
   напрямую. Замена библиотеки/перевод на RTU не должны затрагивать `Poller`/процессоры.

## 7. Точки расширения (дорожная карта доработки)

Здесь то, ради чего документ и передаётся — куда пристыковывать интерфейс и новые функции.

**Интерфейс (UI).** Подписаться на `Poller.SnapshotReady` (или читать `Poller.Latest`) и
рендерить снимок. Для WPF/Avalonia/WinForms/Blazor завести ViewModel, обновляемую по событию.
`Poller.RunAsync` уже асинхронный — гонять его в фоне, маршалить обновления в UI-поток.
Ничего в ядре менять не нужно: UI — ещё один потребитель снимков.

**Хранение истории.** Добавить `ITagProcessor`, пишущий снимки/агрегаты в БестокБД
(PostgreSQL/ClickHouse/файлы). Уже обсуждалась аналитика (Postgres/ClickHouse) — сюда же
ложатся временные ряды тегов и события циклов.

**Кривая цикла (AI1).** `PlatenDistance` при 0,5 с даёт ~10–20 точек на цикл. Для построения
кривой — накапливать сэмплы между границами цикла (по фронтам DI) в процессоре и отдавать
завершённую кривую в UI/БД. Для детальной кривой шаг опроса придётся уменьшать — упрёмся в
реальный отклик устройства (измерять).

**Контроль качества по подушке (AI2).** Уже есть заготовка `CushionStabilityProcessor`:
замер по спаду фазы впрыска, разброс в скользящем окне. Развить до трендов, порогов алярма,
статистики (SPC-карты).

**Детекция циклов и статистика.** `EdgeCycleProcessor` считает факт цикла по фронту DI с
мёртвым временем. Развить: длительность цикла (время между событиями), фазовая
последовательность (закрытие → впрыск → выталкивание → открытие), детекция аномального
порядка/пропущенных фаз.

**Усреднение/сглаживание аналога.** Значение АЦП «дрожит». Добавить процессор скользящего
среднего/медианы для `PlatenDistance`/`Cushion` перед логированием и сравнением с порогами
(с гистерезисом).

**Алярмы.** Процессор, сравнивающий теги с порогами (с гистерезисом для аналога) и
поднимающий события. UI/уведомления подписываются на них.

**Несколько устройств.** Сейчас один `Poller` на устройство. Для парка ТПА — по `Poller` на
устройство, общий агрегатор снимков. Конфиг можно расширить до списка устройств.

**Конфиг во время работы.** Пока конфиг читается один раз при старте. При необходимости —
hot-reload с пересборкой `BatchPlanner` и переинициализацией `Poller`.

## 8. Известные ограничения (осознанные)

- **Только линейный пересчёт** (`scale`/`offset`). Произвольные формулы-выражения намеренно
  не заведены (нет движка выражений). Для нелинейного датчика — либо добавить процессор, либо
  ввести опциональный `formula` с NCalc/DynamicExpresso (тогда — как ещё одна ветка декода,
  не смешивая со `scale/offset`).
- **Многорегистровые значения (`count>1`)** декодер сейчас берёт по первому слову. Для 32-бит
  величин (timestamp) нужно добавить сборку слов с учётом `wordOrder` — под текущие 6 тегов
  (все `count=1`) не требуется.
- **Один Modbus-адрес на опросчик.** Каскад RTU-устройств за USR (master mode) не
  поддерживается — не нужен для задачи.
- **Компиляция/restore не проверялись в песочнице** (нет доступа к nuget.org и SDK при
  генерации). NModbus подтянется при `dotnet restore`. Версия `3.0.*` — при желании
  зафиксировать.
- **`EdgeCycleProcessor` привязан к `MoldOpen`** (фронт «открытие формы») в `Program.cs`.
  Если по реальной схеме завершение цикла удобнее ловить на другом сигнале/спаде — менять имя
  тега и `risingEdge`, ядро не трогать.

## 9. Зависимости и запуск

- **.NET 9**, C# latest, nullable enabled.
- **NModbus** `3.0.*` (Modbus TCP клиент).
- Запуск: `dotnet run` из каталога проекта. `registers.json` копируется в вывод сборки.
- Перед первым запуском на железе: сверить `host`, `port` (регистр `0x1076`), `unitId`.

## 10. Краткая инструкция для Claude Code

При доработке держать в голове: устройство read-only и специфично по функциональным кодам
(§2); граница stateless/stateful — несущая (§3, §6); UI и новые функции подключаются как
потребители снимков и как `ITagProcessor`, без переписывания ядра (§7). Прикладная область —
контроль цикла литья: DI отслеживают фазы, AI дают показатели качества (подушка) и кривую
цикла.
