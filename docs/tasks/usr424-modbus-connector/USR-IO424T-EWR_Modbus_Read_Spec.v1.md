# USR-IO424T-EWR — спецификация Modbus для read-only клиента (C#)

Документ описывает всё, что нужно для программы, которая **только считывает** данные с
USR-IO424T-EWR по Modbus. Основан на User Manual V1.0.3.01.

---

## 1. Что физически есть на устройстве

| Тип | Кол-во | Диапазон / формат | Примечание |
|---|---|---|---|
| Дискретные входы (DI) | 4 | замыкание DI↔COM | по умолчанию «сухой контакт» |
| Релейные выходы (DO) | 4 | AC 220V 5A / DC 28V 5A | пассивное реле, общий COM |
| Аналоговые входы (AI) | 2 | 0–10 В (по умолч.) или 0–20 мА | режим тока — только под заказ |
| Температура (PT100) | 1 | −100…+200 °C | трёхпроводная схема |

Для read-only клиента интерес представляют DI, AI, температура, счётчики импульсов и
(опционально) обратное чтение состояния реле.

---

## 2. Основы Modbus для этого устройства

- **Протоколы:** Modbus RTU (по RS485) и Modbus TCP (по сети). Внутри устройство
  конвертирует принятый Modbus TCP в Modbus RTU и обрабатывает его на MCU.
- **Базовый адрес регистров:** `0x0000`.
- **Разрядность:** каждый регистр 16 бит, порядок байт внутри регистра — **big-endian**
  (старший байт первым). Подтверждается примерами из мануала: температура возвращается как
  `06 92` = `0x0692`.
- **Modbus-адрес устройства (Unit ID):** задаётся в `0x00B1`, диапазон `0x0001…0x00FD`.
  В примерах мануала используется **17 (0x11)** — это и есть типовое заводское значение.
- **Важно:** в slave-режиме (значение по умолчанию) устройство **отбрасывает** кадры, чей
  адрес не совпадает с его локальным Modbus-адресом. Unit ID в запросе обязан совпадать.
- **CRC (только для RTU):** стандартный Modbus CRC-16 (`0xA001`). При работе по TCP CRC не
  нужен — вместо него MBAP-заголовок. Любая нормальная библиотека считает CRC/MBAP сама.

## 3. Карта регистров для чтения

Ниже — только то, что имеет смысл читать. Указаны рекомендуемые функциональные коды (FC).

### 3.1. Дискретные входы — состояние (уровень)

| Параметр | Значение |
|---|---|
| Адреса | `0x0020…0x0023` (DI1…DI4) |
| FC | **0x02** (Read Discrete Inputs); по тексту мануала также допустим 0x03 |
| Данные | бит: `1` = DI замкнут с COM, `0` = разомкнут |
| Кол-во | 4 бита |

Пример (Unit ID 0x11, чтение DI1): `11 02 00 20 00 01 …` → `11 02 01 01 …` (замкнут) /
`11 02 01 00 …` (разомкнут).

Это **основной** регистр для мониторинга входов. Читается без побочных эффектов, можно
опрашивать сколь угодно часто.

### 3.2. Дискретные входы — «кнопка» (событие) ⚠️

| Параметр | Значение |
|---|---|
| Адреса | `0x0030…0x0033` (DI1…DI4) |
| FC | 0x03 / 0x04 |
| Данные | `0xFF00` = было нажатие, `0x0000` = нет |

⚠️ **Регистр самоочищается после первого чтения** (сбрасывается в `0x0000`). Для read-only
мониторинга уровня входов **не используйте** этот регистр — вы будете «съедать» события.
Он нужен только если вам осознанно нужна edge-семантика (зафиксировать факт кратковременного
нажатия между опросами). Для обычного контроля состояния используйте `0x0020…0x0023`.

### 3.3. Счётчики импульсов

| Параметр | Значение |
|---|---|
| Адреса | `0x0040…0x0043` (по одному 16-битному счётчику на вход) |
| FC | 0x03 / 0x04 |
| Данные | `uint16`, диапазон 0…65535 |

Особенности: счётчик **переполняется** (после 65535 начинает с 0), **не сбрасывается** ни
чтением, ни перезапуском (при рестарте значение не сохраняется, т.е. обнуляется питанием).
Дребезг **не фильтруется** — считаются все фронты. Для корректного подсчёта приращений между
опросами обрабатывайте wrap-around на стороне ПО (беззнаковая разность).

### 3.4. Температура (PT100)

| Параметр | Значение |
|---|---|
| Адрес | `0x0050` |
| FC | 0x03 / 0x04 |
| Формула | `T(°C) = (raw − 10000) / 100` |

`raw` — беззнаковый 16-битный код (0…30000 для диапазона −100…+200 °C). Пример: `0x0692`
= 1682 → `(1682 − 10000) / 100 = −83.18 °C`. Отрицательным температурам соответствует
`raw < 10000`. Обратите внимание: это **смещённое** представление, а не two’s complement —
читайте `raw` как `ushort` и применяйте формулу.

### 3.5. Аналоговые входы — напряжение

| Параметр | Значение |
|---|---|
| Адреса | `0x0058` (AI1), `0x0059` (AI2) |
| FC | 0x03 / 0x04 |
| Данные | значение **напрямую в мВ** |

Пример: `0x1000` = 4096 → 4096 мВ ≈ 4.096 В. Диапазон 0–10 В ≈ 0…10000 мВ.

### 3.6. Аналоговые входы — ток (только для «токовой» версии) ⚠️

| Параметр | Значение |
|---|---|
| Адреса | `0x0060…0x0061` |
| FC | 0x03 / 0x04 |
| Данные | 0…20 мА |

⚠️ По умолчанию устройство в режиме измерения **напряжения**; регистры тока в этом случае
следует **игнорировать** (значение неинформативно). Токовый режим поставляется только под
заказ.

### 3.7. Обратное чтение состояния реле (опционально)

| Параметр | Значение |
|---|---|
| Адреса | `0x0000…0x0003` (DO1…DO4) |
| FC | **0x01** (Read Coils) |
| Данные | бит: `1` = реле замкнуто (DO↔COM), `0` = разомкнуто |

Read-only клиенту это позволяет наблюдать текущее состояние выходов, не управляя ими.

### 3.8. Время / RTC (опционально)

| Параметр | Адрес | FC | Формат |
|---|---|---|---|
| Unix-timestamp | `0x0068…0x0069` | 0x03 / 0x04 | 32 бита (2 регистра) |
| Дата/время | `0x006A…0x006C` | 0x03 / 0x04 | BCD: год,мес,день,час,мин,сек |
| День недели | `0x006D` | 0x03 / 0x04 | `0x0001…0x0007` = Пн…Вс |

Для 32-битного timestamp порядок слов (старшее/младшее слово первым) в мануале явно не
задан — при использовании проверьте эмпирически.

### 3.9. Версии/диагностика (опционально)

| Параметр | Адрес | Формат |
|---|---|---|
| MCU software version | `0x00B4` | `0x0112` = V1.1.2 |
| MCU hardware version | `0x00B5` | `0x0110` = V1.1 |

## 4. Рекомендуемая стратегия опроса

Карта регистров **разрывная**, поэтому не читайте один гигантский блок «от 0x0000 до 0x0060».
Разбейте на несколько адресных блоков, каждый — отдельным запросом. Минимальный набор для
типового мониторинга:

| # | Что | FC | Старт | Кол-во | Обязателен? |
|---|---|---|---|---|---|
| 1 | DI1…DI4 (уровень) | 0x02 | `0x0020` | 4 бита | да |
| 2 | Температура | 0x04 | `0x0050` | 1 рег | если нужна |
| 3 | Напряжения AI1/AI2 | 0x04 | `0x0058` | 2 рег | если нужны |
| 4 | Счётчики импульсов | 0x04 | `0x0040` | 4 рег | если нужны |
| 5 | Состояние реле | 0x01 | `0x0000` | 4 бита | опц. |

Можно объединить п.2 и п.3 в один запрос `0x0050…0x0059` (10 регистров): вы прочитаете
температуру, затем 7 «резервных» регистров (значение неопределённо — игнорируйте), затем два
напряжения. Это экономит один round-trip ценой чтения мусорных регистров между ними. Что
выгоднее — зависит от того, критична ли задержка; для малого числа запросов проще держать
блоки раздельными и читаемыми.

FC 0x04 (Read Input Registers) и FC 0x03 (Read Holding Registers) для этих данных
взаимозаменяемы. Для семантической чистоты «только чтение измерений» логичнее 0x04.

## 5. Подводные камни (обязательно учесть)

1. **Кнопочные регистры `0x0030…0x0033` самоочищаются при чтении.** Никогда не опрашивайте их
   в цикле, если вам нужен просто уровень входа — используйте `0x0020…0x0023`.
2. **Счётчики импульсов переполняются на 65535** и не сбрасываются. Обрабатывайте wrap на ПО.
3. **Температура — смещённое кодирование** `(raw − 10000) / 100`, не two’s complement.
   Отрицательные значения дают `raw < 10000`.
4. **Токовые регистры `0x0060…0x0061` неинформативны** на устройстве в режиме напряжения
   (стандарт по умолчанию). Не декодируйте их без явной токовой версии.
5. **Unit ID обязан совпадать** с Modbus-адресом устройства (`0x00B1`, обычно 17). Иначе
   устройство молча отбросит запрос в slave-режиме.
6. **TCP-порт нестандартен по умолчанию.** Порт прослушивания в режиме LAN задаётся в
   `0x1076`. Классический Modbus TCP — 502, но у этого устройства значение конфигурируемое;
   **уточните фактический порт** (через web-конфигуратор / setup-софт) перед подключением.
7. **Разрывная карта регистров** — не читайте несуществующие/резервные адреса «оптом» без
   понимания, что вернётся. Резервные регистры возвращают неопределённые данные.
8. **Задержка отклика в спецификации не указана.** Устройство делает конвертацию TCP→RTU и
   имеет отдельные MCU и коммуникационный модуль, поэтому реалистичное время round-trip —
   скорее десятки миллисекунд, чем единицы. Для быстрых сигналов период обновления самих
   входов тоже неизвестен (в мануале явно сказано, что вход не фильтруется и требует
   стабильной формы сигнала). Если критична частота — измерьте на реальном железе.
9. **Режимы устройства.** Для этого сценария нужно: рабочий режим `0x00B2` = Modbus
   (`0x0001`), RS485-режим `0x00B0` = Slave (`0x0002`, значение по умолчанию). Управлять этим
   из read-only клиента не нужно — просто убедитесь, что устройство в этих режимах.

## 7. Рекомендации по реализации на C#

**Библиотека.** Для .NET 9 подойдут:
- **FluentModbus** — современная, актуальная под свежие .NET, поддерживает Modbus TCP и RTU.
- **NModbus** — зрелая, широко используется; удобные методы, названные по операциям.

**Соответствие FC ↔ методам библиотеки:**

| FC | Операция | Метод (типичное имя) |
|---|---|---|
| 0x01 | Read Coils | `ReadCoils` |
| 0x02 | Read Discrete Inputs | `ReadDiscreteInputs` |
| 0x03 | Read Holding Registers | `ReadHoldingRegisters` |
| 0x04 | Read Input Registers | `ReadInputRegisters` |

**Адресация.** Передавайте адрес как 0-based `ushort`, численно равный адресу регистра:
`0x0020` = 32, `0x0050` = 80, `0x0058` = 88 и т.д. Учтите, что некоторые библиотеки/утилиты
используют «человеческую» нумерацию с +1 или префиксами (3xxxx/4xxxx) — сверяйтесь с
документацией конкретной библиотеки, чтобы не промахнуться на 1.

**Порядок байт.** Каждый регистр big-endian. Многие .NET-библиотеки отдают сырые слова —
при чтении `ushort` проверьте, не нужно ли поменять байты местами (свап), сравнив с известным
примером из мануала (температура `0x0692` = 1682).

**Декодирование (иллюстративно):**

```csharp
// DI (FC02): bool[] по битам
// AI напряжение (FC04, 0x0058): значение уже в мВ
double aiVolt1_mV = raw0x0058;           // напр., 4096 -> 4.096 В

// Температура (FC04, 0x0050)
double tempC = (raw0x0050 - 10000) / 100.0;  // 1682 -> -83.18 °C

// Счётчик импульсов: обработка переполнения между опросами
ushort delta = (ushort)(current - previous); // беззнаковый wrap 65535->0
```

**Опрос и устойчивость:**
- Задайте разумный **таймаут** (например, 500–1000 мс) и **ретраи** (2–3 попытки).
- Не открывайте новое TCP-соединение на каждый опрос — держите одно соединение и
  переоткрывайте только при ошибке.
- Опрашивайте **последовательно** (не шлите новый запрос, не получив ответ на предыдущий):
  устройство — простой slave, конвейерных запросов лучше избегать.
- Логируйте время round-trip — это единственный надёжный способ узнать реальную скорость
  отклика (в спецификации её нет).
- Для нескольких значений группируйте по адресным блокам (см. раздел 4), чтобы уменьшить
  число запросов.

## 8. Итоговый чек-лист для программы (read-only)

- [ ] Подключение: TCP к IP устройства на порт из `0x1076` (уточнён), Unit ID = 17.
- [ ] DI: FC02, `0x0020`, 4 бита.
- [ ] Температура: FC04, `0x0050`, формула `(raw−10000)/100`.
- [ ] Напряжения: FC04, `0x0058`, 2 регистра, значение в мВ.
- [ ] (Опц.) Счётчики: FC04, `0x0040`, 4 регистра, обработка wrap.
- [ ] (Опц.) Реле: FC01, `0x0000`, 4 бита.
- [ ] Не читать кнопочные регистры `0x0030…0x0033` (самоочистка).
- [ ] Big-endian при декодировании, таймаут + ретраи, единичное соединение.
