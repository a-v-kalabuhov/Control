# Резюме проекта: коннектор ТПА BORCHE BM180-MT (USR-IO424T-EWR → Modbus TCP)

Контекст для продолжения работы в новой беседе. Дата: июль 2026.

---

## 1. Задача

Пилотный проект промышленного мониторинга: съём сигналов с термопластавтомата (ТПА)
**BORCHE BM180-MT** с контроллером **KEBA 2980**. Сигналы снимаются через IoT-шлюз
**USR-IO424T-EWR** (4 DI сухой контакт + 2 AI 0–10 В + 1 PT100, Modbus TCP/RTU),
опрашиваются программой-«коннектором» на **C# .NET 9** (read-only, период 500 мс).
Коннектор вычисляет состояние машины и отдаёт его наверх; верхний слой (шаблон устройства,
MQTT/REST наружу) пользователь реализует сам. Код дорабатывается через Claude Code.

## 2. Ключевые факты об устройстве USR-IO424T-EWR

- Slave-режим по умолчанию; Modbus TCP конвертируется внутри в RTU. Unit ID обычно 17.
  TCP-порт конфигурируемый (регистр 0x1076) — сверять с фактическим.
- Регистры: DI уровень `0x0020–0x0023` (FC02); «кнопка» `0x0030–0x0033` (FC03/04,
  **самоочистка при чтении** — не использовать для уровня); счётчики импульсов
  `0x0040–0x0043` (uint16, wrap 65535, нет сброса, нет антидребезга); температура `0x0050`
  (формула `(raw−10000)/100`); напряжение AI1/AI2 `0x0058/0x0059` (значение в мВ);
  реле `0x0000–0x0003` (FC01). Big-endian.
- Время отклика в доке не указано; для 500 мс запас большой, для быстрых опросов — мерить.
- DI — сухой контакт (детекция замыкания DI↔COM, не напряжения). Потенциальные сигналы
  (24 В) подключаются только через оптопару.

## 3. Электрика машины (из альбома схем 1430X006, 33 листа)

- **Транзисторные выходы +24 В** (нужна оптопара): командные DO KEBA 2980 (Mold close DO3,
  Injection 1 DO14, Plasticize 1 DO15…), статусы модуля DO570 на его клеммах.
- **Сухие контакты (напрямую на USR)** — реле роботного интерфейса EU12 (лист 27):
  Mould closed → **KA23**, Mould openend (DO570 DO1, провод 158) → **KA24** (в разъёме
  робота пин зовётся «Mould open position» — на этой машине совпадают из-за уставки полного
  хода), Ejector BWD/FWD reached → **KA26/KA27**, Reject → **KA34**, Intermediate opening →
  **KA37**. Проверить наличие свободных контактных групп реле — не вмешиваться в цепь робота.
- **Аналог**: датчики положения **Miran LS2** — резистивные, ratiometric, выход 0…V питания;
  здесь UREF=10 В → 0–10 В однополярно. Mold Position = LT1 (450 мм, AI0), Injection1
  Position = LT3 (225 мм, AI2). Пересчёт: мм = (V/10)×ход → scale 22.5 и 45 мм/В.
  Врезка высокоомная/через изолирующий сплиттер. Для стабильности подушки можно оставаться
  в вольтах.
- Машина сервонасосная (INOVANCE 15 кВт), группа клапанов впрыска DO13/14/15 сблокирована
  через KA20 (общий гидропоток).

## 4. Предметная область (согласованная терминология)

- **Цикл** (по панели KEBA «Время действия»): Закр ПФ → Сопло вперёд → Впрыск (включает
  выдержку — отдельным этапом не выделяется) → Набор → Декомпр → Охлажде → Сопло назад →
  Откр ПФ → Толкат вперёд → Толкат назад. Панель ведёт эталон времён этапов («УстЭтал»).
- **Подушка (cushion)** — остаток расплава перед шнеком в конце выдержки = минимум позиции
  шнека за цикл. Ключевой показатель стабильности процесса (дрейф/разброс → износ обратного
  клапана и т.п.). **Замер = минимум AI2 за окно цикла**, НЕ мгновенное значение (к E1 шнек
  уже набрал новую дозу; точный момент спада Injection при поллинге не виден).
- **Reject** (DO570 DO5) — вердикт контроллера «деталь брак», окно выталкивания.
- **Граница цикла**: формально по KEBA — «Толкат назад» (E2); для мониторинга рабочая
  граница — **Ejector FWD reached (E1)** = «деталь изготовлена».
- Уставка = setpoint; доза задаётся позицией шнека (сечение цилиндра постоянно).

## 5. Автомат состояния (главные согласованные решения)

- **Якорь цикла — впрыск** (`Injection ↑`), не Mould closed (форма смыкается и вручную).
  Детекция — **дискретный сигнал** Injection 1 (DO14 через оптопару), не порог по аналогу
  (порог требует технологической калибровки — overengineering).
- **`CycleCounter` = счётчик износа формы**: +1 на фронте впрыска, независимо от исхода.
- `mode ∈ {idle, auto, alarm, offline}` — **manual убран** (без впрыска наладку от простоя
  не отличить). Idle/alarm взаимоисключающие: был впрыск → только alarm (E1 не пришёл);
  не было → только idle. `cycleCompletion ∈ {NoneYet, Normal, Aborted, Interrupted}`.
- Таймауты **адаптивные**: скользящее среднее последних N успешных auto-циклов ×
  коэффициенты (alarmCoef, idleCoef); seed 300 000 мс; сброс статистики при простое >15 мин
  (переналадка). Offline — N подряд неудачных опросов (демпфер от ложных Interrupted).
- **Принцип: коннектор снимает, интерпретируют внешние системы** — подушка публикуется при
  любом исходе цикла.
- `MachineState` (публичный контракт): строгие Mode/CycleCounter/LastCycleCompletion/
  Timestamp + гибкий словарь `Fields` (cushion, cushion2, lastCycleDurationMs, reject,
  moldPosition, injectionPosition, поля шаблона).
- **Семантические роли** (`SignalRole`): Injection, EjectorFwdReached, Reject, MouldClosed,
  MouldOpened, EjectorBwdReached, InjectionPosition, MoldPosition, Injection2,
  InjectionPosition2, None (прозрачный проброс). Логика видит сигналы только через роли.
- Профили: `singleNode` (обязательны Injection + EjectorFwdReached), `twoNode`
  (+ InjectionPosition2; вторая подушка — минимум за то же окно, узлы 2K синхронны).

## 6. Артефакты (репозиторий `/mnt/user-data/outputs/UsrConnector/`)

Решение .NET 9: **UsrConnector.Core** (Config/ConfigLoader с ролями и валидацией профилей,
IModbusReader+NModbusReader (только FC01–04), BatchPlanner, RegisterDecoder, RoleMapper,
SignalRoles/RoleSnapshot, MachineState, StateMachineSettings, MachineStateMachine,
ConnectorEngine), **UsrConnector.Core.Tests** (~25 тестов: исполняемая спецификация автомата
+ физический слой), **UsrConnector.Host** (консольный вывод; сюда пользователь добавит
MQTT/REST; connector.json пилота с ролями и калибровкой).

Документы в репозитории: **CLAUDE.md** (входная точка Claude Code, 6 инвариантов: read-only;
Core не знает внешнего транспорта; stateless/stateful граница; роли, не адреса; автомат =
код+тесты+STATE_MACHINE.md синхронно; снимать-не-интерпретировать), **STATE_MACHINE.md**
(спецификация автомата с таблицей переходов), **ARCHITECTURE.md**, **CYCLOGRAM_SINGLE_NODE.md**
(выверенная: листы + панель KEBA + подтверждения; фазы A1–E2 с этапами KEBA, три таблицы
сигналов×фаз, точки съёма §6, детекция фаз §7), **CYCLOGRAM_FULL_TWO_NODE.md** (2K +
поворотный стол + знаки Core 1..4; последовательность узла 2 — модель, требует панели
2-узловой машины), **USR-IO424T-EWR_Modbus_Read_Spec.md**.

Старый проект `/mnt/user-data/outputs/UsrIoPoller/` — предыдущая итерация (generic-опросчик
с ITagProcessor), заменён UsrConnector.

## 7. Открытые вопросы / проверки на месте

1. `dotnet build && dotnet test` — код не компилировался (нет SDK в среде генерации).
2. Фактический TCP-порт USR (регистр 0x1076) и Unit ID.
3. Удержание усилия смыкания: замер DO3 мультиметром в цикле (методика в
   CYCLOGRAM_SINGLE_NODE.md §8.1); влияет только на справочную точность циклограммы.
4. Свободные контактные группы реле KA23/KA27/KA34/KA24 — для съёма без вмешательства
   в цепь робота.
5. Сигналы аварии KEBA (для alarm по событию, не только по таймауту) — нужна документация
   оператора; отложено.
6. Для 2K: панель «Время действия» двухузловой машины (подтвердить последовательность узла
   2 и поворотного стола); дефицит аналоговых входов (2 подушки съедают оба AI) — второй
   модуль USR или отказ от MoldPosition.
7. Порядок DI в connector.json подогнать под фактический монтаж.
