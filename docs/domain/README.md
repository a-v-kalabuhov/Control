# Предметная область Wintime Control

Это справочник **предметной области** (домена) — единый словарь и описание сущностей
производства на термопластавтоматах. Отвечает на вопрос «**что это за понятие и как оно
ведёт себя по правилам бизнеса**», а не «как устроен код».

## Зачем эта папка (и чем отличается от ADR / CLAUDE.md)

У документации три слоя — не путать их:

| Вопрос | Где живёт |
|---|---|
| **Что это за понятие? Как оно ведёт себя по бизнес-правилам?** (факт, словарь, без «решали/отвергли») | **Здесь, `docs/domain/`** |
| **Почему мы так решили и какие альтернативы отвергли?** (решение, трудно откатить) | **ADR** — [`docs/adr/`](../adr/) |
| **Железное правило, нарушение = баг** (сжатый итог) | **CLAUDE.md** (+ указатель сюда / в ADR) |
| **Не сломай вот это при правке** (локальный сигнал) | **Комментарий в коде** (+ ссылка на ADR) |

Грубая эвристика: если можешь начать фразу со слов «**мы решили…, потому что…, отвергнув…**» —
это ADR. Если «**Х — это…, у него есть…, он переходит из… в…**» — это сюда.

## Как описывать сущность

Скопируй [`_TEMPLATE.md`](_TEMPLATE.md), назови файл по сущности (`mold.md`, `event.md` …),
заполни разделы. Не пиши то, что и так видно в коде типом поля — пиши **доменный смысл**
поля и **правила** поведения. Образцы заполнения: [`imm.md`](imm.md), [`shift-task.md`](shift-task.md).

## Единый словарь (ubiquitous language)

Один и тот же объект называем одинаково везде — в разговоре, UI, коде. RU-термин ↔ код:

| Термин (RU) | В коде | Короткий смысл |
|---|---|---|
| ТПА (термопластавтомат) | `Imm` | Литьевая машина — главный объект мониторинга |
| Пресс-форма, ПФ | `Mold` | Сменная оснастка, формирует деталь; ставится на ТПА |
| Сменное задание, СЗ | `ShiftTask` | Производственное задание: что лить, сколько, кем |
| Наладчик | `User` (`Role=Adjuster`) | Ставит ПФ, налаживает ТПА, ведёт задание с планшета |
| Начальник цеха / менеджер | `User` (`Role=Manager`) | Выдаёт задания, ведёт справочники, смотрит отчёты |
| Цикл / смыкание | `ImmCycle` | Один цикл литья (смыкание→впрыск→размыкание) |
| Гнёздность | `Mold.Cavities` | Сколько деталей формируется за один цикл |
| Выпуск / выработка | `ShiftTask.ActualQuantity` | Фактически произведённое количество годных деталей |
| Наладка | `TaskStatus.Setup` | Фаза подготовки ТПА к запуску задания |
| Простой | `Event` (`EventType=Downtime`) | Остановка оборудования с причиной |
| Причина простоя | `DowntimeReason` | Классификатор остановок (плановый/аварийный) |
| Статус ТПА | `ImmStatusHistory.Status` | Сырой режим: Auto/Manual/Idle/Alarm/Offline |
| Эффективное состояние | `ImmEffectiveStatus` | Вычисляемое состояние для дашборда (не хранится) |
| Шаблон оборудования | `Template` | Конфигурация набора датчиков ТПА |
| Смена | `Shift` | Рабочая смена (расписание) |
| Ресурс ПФ | `Mold.MaxResourceCycles` | Предельное число смыканий до выработки |
| ТО-1 / ТО-2 | `Mold.To1Cycles` / `To2Cycles` | Пороги планового обслуживания по числу смыканий |
| Артикул (для QR) | `Mold.FormId` | Уникальный код ПФ, зашит в QR |
| Литник | `Mold.RunnerWeightGrams` | Технологический отход (вес литниковой системы) |

> Заметил расхождение «как в цеху говорят» ↔ «как в коде» — внеси сюда строку. Словарь
> ценен только пока полон.

## Связанные артефакты

- **Матрицы состояний**: [`docs/details/`](../details/) (напр. `Состояния_ТПА.xlsx`) — табличный
  источник правды по переходам режимов/состояний.
- **ADR**: [`docs/adr/`](../adr/) — почему приняты ключевые доменные решения
  (`0001` снапшот Cavities, `0002` КА задания в сущности, `0004` единый источник роли,
  `0006` эффективное состояние и авто-простои).
- **Инварианты-итоги**: [`CLAUDE.md`](../../CLAUDE.md), раздел «Инварианты домена».
