# Сменное задание — СЗ (`ShiftTask`)

> Производственное задание: на каком ТПА, какой пресс-формой, кто и сколько деталей делает.
> Центральная сущность, связывающая ТПА, ПФ, наладчика и факт выработки.

## Что это

Сменное задание (СЗ) — единица планирования и учёта производства. Менеджер создаёт СЗ
(ТПА + пресс-форма + плановое количество), наладчик принимает его на планшете, налаживает
машину и ведёт до завершения. По ходу выполнения система привязывает к заданию циклы
([`imm-cycle.md`](imm-cycle.md)) и считает фактический выпуск.

Задание — носитель жизненного цикла производственного процесса: его статусы определяют, что
сейчас происходит с машиной (наладка идёт / идёт работа / завершено).

## Код

- Сущность: `Wintime.Control.Core.Entities.ShiftTask`
- Статусы: `Wintime.Control.Core.Enums.TaskStatus`
- Конечный автомат — методы прямо в сущности (`Issue`, `StartSetup`, `CompleteSetup`,
  `CancelSetup`, `VerifyMold`, `Complete`, `Close`). См. ADR-0002.

## Ключевые поля

| Поле | Смысл и правила |
|---|---|
| `ImmId` / `MoldId` | Какая машина и какая пресс-форма задействованы |
| `PersonnelId` | Наладчик задания (ссылка на `User.Id`) |
| `PlanQuantity` | Плановое количество годных деталей |
| `ActualQuantity` | Фактический выпуск; фиксируется при завершении |
| `ActualMaterialWeightGrams` | Фактический расход материала |
| `CloseReason` | Причина отклонения от плана / закрытия (см. правила ниже) |
| `MoldVerifiedAt` | Метка успешной QR-верификации ПФ при наладке (без неё — предупреждение) |
| метки `*At` | Время каждого перехода (`IssuedAt`, `SetupStartedAt`, `StartedAt`, …) — основа отчётов по времени наладки/работы |

## Жизненный цикл (конечный автомат)

```
Draft → Issued → Setup → InProgress → Completed → Closed
                   └──── CancelSetup ────┘ (Setup → Issued)
```

| Переход | Метод | Смысл |
|---|---|---|
| `Draft → Issued` | `Issue()` | Менеджер выдал задание наладчику |
| `Issued → Setup` | `StartSetup()` | Наладчик начал наладку (ставит ПФ, настраивает) |
| `Setup → Issued` | `CancelSetup()` | Отмена наладки — сбрасывает `SetupStartedAt`/`MoldVerifiedAt` |
| `Setup` (без смены статуса) | `VerifyMold()` | Зафиксировать QR-верификацию ПФ |
| `Setup → InProgress` | `CompleteSetup()` | Наладка завершена, пошёл серийный выпуск (`StartedAt`) |
| `InProgress → Completed` | `Complete(qty, reason)` | Задание выполнено, зафиксирован факт |
| любой `→ Closed` | `Close(reason)` | Ручное закрытие (конец смены/дня) |

Каждый переход проверяет исходный статус и при нарушении кидает `DomainException` → HTTP 400
(см. ADR-0003). Побочные эффекты (перевод режима ТПА) — в контроллере, не в сущности.

## Инварианты (нарушение = баг)

- **Причина (`CloseReason`) фиксируется только при расхождении факта с планом** —
  `ActualQuantity != PlanQuantity` (как недовыполнение, так и перевыполнение). При совпадении
  причина не нужна. *Почему:* причина-отклонение бессмысленна, когда план выполнен ровно.
- **Переходы только через методы сущности**, не присваиванием `Status` напрямую — иначе
  обойдём проверку исходного статуса и метки времени. → ADR-0002.
- **Статус задания участвует в эффективном состоянии ТПА** (Setup → «Наладка»,
  InProgress → «Работа»). Рассинхрон статуса задания и режима ТПА = «работа без задания».
  → [`imm-status.md`](imm-status.md), ADR-0006.
