**Спецификация требований к программному комплексу "CONTROL: Управление цехом ТПА"**

**г. Москва** **2026 г.**

**1\. Введение**

**1.1. Наименование программы**

Полное наименование: Программное обеспечение для контроля и управления цехом термопластавтоматов (ТПА) "CONTROL".  
Краткое наименование: ПО "CONTROL".

**1.2. Назначение разработки**

ПО "CONTROL" предназначено для автоматизации сбора данных с оборудования (ТПА), учета работы пресс-форм и персонала (наладчиков), а также формирования производственных отчетов. Система призвана повысить эффективность управления цехом за счет оперативного контроля производственных показателей и снижения рутинной нагрузки на руководителя.

**1.3. Цели создания системы**

- Автоматизация сбора данных о работе ТПА и пресс-форм.
- Повышение достоверности и оперативности учета выпуска продукции.
- Создание единой базы данных оборудования, оснастки и персонала.
- Обеспечение руководителя инструментами для анализа эффективности работы оборудования и персонала.
- Цифровизация процесса выдачи сменно-суточных заданий (ССЗ).

**2\. Требования к системе в целом**

**2.1. Требования к архитектуре**

- Система должна иметь клиент-серверную архитектуру.
- Доступ к системе должен осуществляться через веб-интерфейс (Web-клиент) с персональных компьютеров.
- Для мобильных рабочих мест (наладчиков) должен быть разработан web-интерфейс адаптированный под разрешение планшетов (как отдельное приложение).
- Серверная часть должна функционировать в локальной сети предприятия (on-premise) с возможностью дальнейшего расширения.

**2.2. Требования к разграничению прав доступа**

Система должна поддерживать ролевую модель доступа (не менее 4 ролей):

1.  **Администратор:** Полный доступ ко всем модулям, включая настройку справочников, управление пользователями и правами.
2.  **Руководитель / Диспетчер:** Доступ к модулям "Справочники" (просмотр), "Модуль диспетчера" (выдача заданий), "Главная страница" (мониторинг), "Отчеты" (все виды).
3.  **Наладчик:** Доступ только к мобильному приложению для просмотра заданий, сканирования QR-кодов и взаимодействия с системой.
4.  **Оператор / Наблюдатель (опционально):** Доступ только к "Главной странице" для просмотра статусов оборудования.

**2.3. Требования к технологиям**

- **Веб-интерфейс:** Корректное отображение в браузерах последних версий (Chrome, Yandex.Browser).
- **Серверная часть:** Языки программирования: C# ([ASP.NET](https://asp.net/) Core).
- **База данных:** Реляционная СУБД (PostgreSQL или MSSQL).

**3\. Функциональные требования**

**3.1. Модуль "Справочники"**

Интерфейс для управления нормативно-справочной информацией. Доступен ролям "Администратор" и "Руководитель".

**3.1.1. Справочник "ТПА"**

Поля для ввода:

- **ID ТПА** (уникальный, генерируется автоматически или задается вручную).
- **Наименование** (текст, обязательное поле).
- **Модель** (текст).
- **Инвентарный номер** (текст).
- **Дата ввода в эксплуатацию** (дата).
- **Активен/Не активен** (флаг для вывода из эксплуатации).
- **(Опционально) IP-адрес / параметры подключения** для сбора данных.

Доступные операции:

- **Редактирование** (редактирование имеющихся станков).

**3.1.2. Справочник "Пресс-форма" (ПФ)**

**Ручной ввод:**

- **FormID** (уникальный, генерируется системой для кодирования в QR).
- **Наименование / Артикул** (текст).
- **Количество гнезд** (число, редактируемое).
- **Вес детали** (число, граммы).
- **Вес литника** (число, граммы).
- **Ресурс** (число, допустимое количество смыканий до списания).
- **ТО 1, ТО 2, ...** (число, количество смыканий для уведомления о техобслуживании).
- **Место хранения** (текст, номер ячейки).
- **Путь к чертежу** (текст, UNC-путь или ссылка в локальной сети).
- **Фото** (загрузка файла jpg/png).

**Автоматически собираемые данные:**

- **Время работы на ТПА** (накопленное, по каждому ТПА и общее).
- **Количество смыканий** (накопленное, по каждому ТПА и общее).

Доступные операции:

- **Редактирование** (редактирование имеющихся станков).
- **Добавление**
- **Активен/Не активен** (флаг для вывода из эксплуатации)

**3.1.3. Справочник "Персонал (Наладчики)"**

- **ID** (табельный номер, строка).
- **ФИО** (текст).
- **Квалификация / Разряд** (текст).
- **Активен** (флаг).
- **Учетные данные для входа в мобильное приложение** (логин/пароль).

Доступные операции:

- **Редактирование** (редактирование имеющихся станков).
- **Добавление**
- **Активен/Не активен** (флаг для вывода из эксплуатации)

**3.1.4. Справочник "Причины простоев"**

- **Наименование причины** (Наладка, ППР, Отсутствие сырья, Авария, Отключение эл/энергии и т.д.)
- **Тип** (Плановый / Аварийный).

Доступные операции:

- **Редактирование** (редактирование имеющихся причин простоев).
- **Добавление**
- **Активен/Не активен** (флаг для вывода из эксплуатации)

**3.2. Модуль "Диспетчерская" (Выдача ССЗ)**

Реализуется в виде веб-таблицы (Excel-like) для планирования работ на смену.

- **Отображение заданий:** Строка — одна задача на станок.
- **Добавление задания:** Кнопка "Новое задание", открывается форма для заполнения.
- **Поля строки задания:**
    1.  **ФИО Наладчика:** Выпадающий список из справочника "Персонал".
    2.  **Оборудование (ТПА):** Выпадающий список из справочника "ТПА" (только активные).
    3.  **Артикул (Пресс-форма):** Выпадающий список из справочника "ПФ" (с подгрузкой атрибутов).
    4.  **План (шт.):** Числовое поле, вручную.
    5.  **Примечание:** Текстовое поле.
    6.  **Действие:** Кнопка "Выдать задание" (отправляет задание в планшет наладчику).

- **Сортировка и фильтрация:** По любому столбцу (по времени выдачи, статусу, ФИО, оборудованию).
- **Индикация статуса задания:**
    1.  **Черновик** (задание на стадии заполнения, в этом статусе можно редактировать и отменять задание).
    2.  **Выдано** (задание отправлено).
    3.  **Наладка** (наладчик принял и сканировал ПФ на ТПА).
    4.  **Выполняется** (ТПА в режиме Auto, счетчик растет).
    5.  **Выполнено** (достигнут план).
- **Прогресс:** Отображение хода выполнения в % (Факт / План \* 100). Факт берется со счетчика ТПА по данной ПФ.

Доступные операции:

- **Редактирование** (редактирование имеющихся причин простоев).
- **Добавление**
- **Активен/Не активен** (флаг для вывода из эксплуатации)

**3.3. Модуль "Мониторинг оборудования" (Главная страница / Дашборд)**

Графическое отображение состояния всех ТПА в реальном времени.

- **Карточки станков:** Каждый ТПА отображается в виде плитки.
- **Основная информация на плитке:**
    - Наименование ТПА.
    - Статус (Auto/Manual/Alarm/Offline) - цветовая индикация.
    - Текущее задание / Пресс-форма.
    - Счетчик циклов (текущий / план).
    - Текущее время цикла.
- **Детальная информация (по клику на плитку):** Всплывающее окно с графиками последних циклов по ключевым параметрам (давление впрыска, позиция шнека, подушка, температуры зон).
- **Тревоги:** Отображение активных аварий по станку.

**3.4. Мобильное WEB-приложение (Планшет Наладчика)**

Функции для роли "Наладчик":

1.  **Авторизация:** Вход по логину/паролю.
2.  **Просмотр заданий:** Список выданных заданий на смену с указанием ТПА, ПФ и плана.
3.  **Прием задания в работу:** Кнопка "Взять задание".
4.  **Сканер QR-кодов:** Встроенный функционал сканирования.
    ЭТИ СЦЕНАРИИ БОЛЕЕ НЕДЕЙСТВИТЕЛЬНЫ!!!!!
    СКАНИРОВАНИЕ НЕ ПРИВЯЗАНО К СМЕНЕ СОСТОЯНИЯ ЗАДАНИЙ.
    СМЕНА СОСТОЯНИЯ ЗАДАНИЙ ВЫПОЛНЯЕТСЯ НАЖАТИЯМИ КНОПОК В ИНТЕРФЕЙСЕ НАЛАДЧИКА.
    - **Сценарий 1: Получение ПФ со склада.** Наладчик сканирует QR-код ячейки или ПФ, подтверждая факт изъятия.
    - **Сценарий 2: Установка ПФ на ТПА.** Наладчик сканирует QR-код ПФ, затем QR-код на ТПА. Система фиксирует: ПФ "FormID" установлена на ТПА "ID", начало периода работы, привязывает текущее задание.
    - **Сценарий 3: Сдача ПФ на склад.** Наладчик сканирует QR-код ПФ и/или ячейки хранения, завершая цикл работы с оснасткой.
5.  **Отчет об окончании работы:** Кнопка "Завершить задание" (с обязательным указанием причины, если план не выполнен).
6.  **Фиксация простоев (опционально):** Возможность отметить начало/конец простоя с выбором причины из справочника, если наладчик отошел от станка.

**3.5. Модуль "Отчеты"**

Генерация аналитических отчетов в веб-интерфейсе с возможностью выгрузки в Excel и PDF.

**3.5.1. Отчет "Картина рабочего дня" (За смену)**

- **Параметры:** Выбор даты, выбор смены (1/2), выбор оборудования (весь цех / конкретный ТПА).
- **Форма вывода:**
    - _График Ганта:_ Визуализация временных отрезков работы (циклы), простоев, выключений по каждому станку.
    - _Таблица:_
        - ТПА / Период
        - Наименование ПФ
        - Заданное время цикла
        - Среднее время цикла
        - Время в работе (сумма циклов)
        - Время простоев
        - Время выключено
        - Количество смыканий
        - Количество годных изделий (смыкания \* гнезда)
        - Эффективность (Работа / (Работа + Простой))
        - Детализация простоев по причинам (если введены)

**3.5.2. Отчет "Производительность оборудования" (За период)**

- **Параметры:** Диапазон дат (неделя, месяц, квартал, год, произвольный).
- **Форма вывода:**
    - _Столбчатая диаграмма:_ Загрузка (Работа/Простой) каждого ТПА по дням.
    - _Сводная таблица:_
        - ТПА
        - Суммарное время работы
        - Суммарное время простоев
        - Общее количество смыканий
        - Средняя эффективность за период

**3.5.3. Отчет "Активы цеха"**

- **По пресс-формам:** Выбор диапазона дат. Таблица: ПФ, Наработка (часы), Количество смыканий, Время под нагрузкой на каждом ТПА, Остаток ресурса.
- **По наладчикам:** Выбор диапазона дат. Таблица: Наладчик, Количество выполненных заданий, Суммарное время работы, Среднее время наладки, Количество отработанных смен, ТПА/ПФ с которыми работал.

**4\. Требования к интеграции и сбору данных**

**4.1. Сбор данных с ТПА**

- Система должна опрашивать контроллеры ТПА с заданной периодичностью (например, 1 раз в 5 секунд) или получать данные по подписке.
- **Основные параметры (телеметрия):**
    - Статус машины (Auto/Manual/Alarm/Offline) - дискретный сигнал.
    - Счетчик циклов (накопленный итог).
    - Время цикла (текущее / среднее).
    - Время впрыска / охлаждения.
- **Вспомогательные параметры (для анализа качества):**
    - Давление впрыска (макс., мин.).
    - Позиция шнека.
    - Подушка (cushion).
    - Температуры по зонам.
    - Код текущей аварии.

**4.2. Управление QR-кодами**

- Генерация QR-кодов для Пресс-форм при их создании.
- Содержимое QR-кода: уникальная ссылка или JSON-объект вида {"entity": "mold", "id": "PF-001"}.
- Привязка QR-кода на ТПА (создается администратором и распечатывается наклейка). Содержимое: {"entity": "machine", "id": "TPA-05"}.
- QR-код ячейки хранения: {"entity": "storage", "id": "A-12"}.

**5\. Требования к интерфейсу**

- Веб-интерфейс должен быть интуитивно понятным и адаптированным под экраны ПК (Full HD).
- Дашборд руководителя должен позволять оценить ситуацию за 10-15 секунд (минимум текста, максимум графики и цветовых индикаторов).
- Мобильное приложение должно быть оптимизировано для работы на экранах 7-10 дюймов с учетом использования в "грязной" среде (крупные кнопки, высокая контрастность).

**6\. Требования к безопасности**

- Хранение паролей в хешированном виде.
- Разграничение доступа на уровне серверной логики.
